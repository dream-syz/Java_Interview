## 〇、什么是云原生 🔥

云原生（Cloud Native）是一套**面向云环境构建和运行可弹性扩展应用**的方法论。CNCF 定义的代表技术：**容器、微服务、服务网格、不可变基础设施、声明式 API**。

### 12-Factor（十二要素）应用 ⭐

云原生应用设计准则，高频的几条：

- **配置与代码分离**：配置放环境变量（对应 K8s 的 ConfigMap/Secret）。
- **无状态进程**：应用不在本地存状态，便于水平扩展与随时销毁（状态externalize 到 DB/Redis）。
- **端口绑定、日志当事件流**（输出到 stdout，由平台采集，见 [25-可观测性](./25-可观测性.md)）。
- **快速启动与优雅终止**、**环境一致性**（开发/测试/生产尽量一致）。

> 一句话：**可弹性、可观测、可自动化、声明式、不可变**。面试讲云原生别只说"用了 Docker/K8s"，要点出这套理念。

## 一、容器与 Docker

### 1. 容器与虚拟机的区别 🔥

| | 容器 | 虚拟机 |
| --- | --- | --- |
| 隔离级别 | 进程级（共享宿主内核） | 硬件级（独立 OS + Hypervisor） |
| 资源开销 | 小（MB 级，秒级启动） | 大（GB 级，分钟级启动） |
| 性能 | 接近原生 | 有虚拟化损耗 |
| 隔离/安全 | 较弱（共享内核） | 强 |
| 密度 | 单机可跑成百上千 | 数十 |

> 二者不对立：生产常**「虚拟机/裸金属上跑 K8s，K8s 里跑容器」**，兼顾隔离与密度。

### 2. 容器隔离的底层原理 🔥

容器本质是**宿主机上一个被隔离和限制的普通进程**，靠 Linux 内核三大机制：

- **Namespace（命名空间）**：隔离**视图**。常见 6+ 种：`PID`（进程）、`Network`（网卡/端口）、`Mount`（文件系统）、`UTS`（主机名）、`IPC`、`User`（用户/权限）。让容器以为自己独占系统。
- **Cgroups（控制组）**：限制**资源**配额（CPU、内存、磁盘 IO、网络带宽），超限会被限流或 OOM Kill。
- **UnionFS（联合文件系统，如 overlay2）**：实现镜像**分层**与**写时复制**。

> 记忆口诀：**Namespace 管"看得见什么"（隔离），Cgroups 管"能用多少"（限制），UnionFS 管"镜像怎么存"。**

### 3. 镜像分层、写时复制与 OCI 规范 🔥

- 镜像由多个**只读层**叠加，**每条 Dockerfile 指令产生一层**，层间通过哈希标识，可被多个镜像**共享复用**（节省存储、加速拉取/构建）。
- 容器启动时在镜像顶部加一个**可写层（容器层）**，修改文件时**写时复制（CoW）** 到可写层，不污染底层镜像；容器删除则可写层丢失（所以数据要挂 Volume）。
- **OCI（Open Container Initiative）** 标准化了镜像格式（image-spec）和运行时（runtime-spec），使 Docker、containerd、Podman 等可互通，镜像不绑定 Docker。

### 4. Dockerfile 编写与镜像优化 🔥（实战）

**核心目标：镜像更小、构建更快、更安全。** Java 应用的多阶段构建范例：

```dockerfile
# ---- 构建阶段（用完整 JDK + Maven）----
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn -B dependency:go-offline      # 先下依赖，利用层缓存（pom 不变就不重下）
COPY src ./src
RUN mvn -B clean package -DskipTests

# ---- 运行阶段（只要 JRE，体积小）----
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
RUN addgroup -S app && adduser -S app -G app   # 非 root 运行，提升安全
USER app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

优化要点：

- **多阶段构建**：编译用大镜像，运行只拷产物到精简镜像，体积可从 700MB+ 降到 200MB 内。
- **善用层缓存**：变动少的指令放前面（先 `COPY pom.xml` 下依赖，再 `COPY` 源码），改代码时不必重下依赖。
- **精简基础镜像**：`alpine`（小但用 musl libc，偶有兼容问题）、`distroless`（无 shell、攻击面小）。
- **减少层数、清理缓存**：合并 `RUN`，删包管理缓存。
- **`.dockerignore`**：排除 `target`、`.git`、`node_modules`，减小构建上下文。
- **非 root 用户**运行、固定基础镜像版本（别用 `latest`）。

### 5. Java 应用容器化的坑（JVM 容器感知）🔥🔥（高频且实战）

老版本 JDK（8u191 之前）**感知不到 cgroup 限制**，会按宿主机的 CPU/内存来设默认堆和线程池，导致容器内存超 limit 被 **OOM Kill**、线程数爆炸。

- **JDK 10+ / 8u191+** 默认开启 `UseContainerSupport`，能正确读取 cgroup 的 CPU/内存配额。
- 推荐用**比例**而非固定值设堆：`-XX:MaxRAMPercentage=75.0`（堆占容器内存 75%，留出元空间/线程栈/直接内存）。
- 注意 **CPU limit 影响 GC/JIT 线程数、`Runtime.availableProcessors()`、线程池大小**，CPU 给太少会拖慢。
- 容器内存 = 堆 + 元空间 + 线程栈 + 直接内存 + JVM 自身，**limit 要留足非堆开销**，否则 RSS 超限被杀（详见 [03-JVM](./03-JVM.md)）。

### 6. 镜像安全 ⭐

- **最小化基础镜像**（distroless / alpine），减少攻击面。
- **非 root 运行**、只读根文件系统、丢弃多余 Linux capabilities。
- **镜像扫描**：Trivy / Clair 扫描 CVE 漏洞，集成到 CI 卡点。
- **镜像签名与验签**（Cosign / Notary），保证来源可信。
- 不在镜像里打包密钥（用 Secret / 外部密钥管理）。

### 7. 容器运行时的演进 ⭐

- 早期 K8s 通过 **dockershim** 调 Docker，链路长（kubelet → dockershim → dockerd → containerd → runc）。
- **K8s 1.24 起移除 dockershim**，直接通过 **CRI（Container Runtime Interface）** 对接 **containerd / CRI-O**，链路更短。
- 说明：**Docker 构建的镜像仍能在 K8s 跑**（都遵循 OCI），只是运行时不再用 dockerd。
- `runc`（OCI 运行时）真正创建容器；安全沙箱场景用 gVisor / Kata Containers。

### 8. 镜像仓库 ⭐

存储和分发镜像。公有：Docker Hub；私有自建：**Harbor**（企业首选，支持权限、镜像扫描、签名、复制、配额）。CI 构建完推到仓库，K8s 从仓库拉取。

### 9. 常用命令 ⭐

`docker build -t app:1.0 .`、`run`、`ps`、`images`、`exec -it`、`logs -f`、`rm`/`rmi`、`tag`/`push`/`pull`；多容器本地编排用 `docker compose up`。

## 二、Kubernetes（K8s）

### 10. 为什么需要 K8s 🔥

Docker 解决"单个容器怎么跑"，K8s 解决"**大规模容器集群怎么编排管理**"：自动调度、自愈（探针+重启）、弹性伸缩、滚动发布与回滚、服务发现与负载均衡、配置与密钥管理、声明式编排。

### 11. 架构（控制面 + 数据面）🔥

**Master / 控制面（Control Plane）**：

- **API Server**：唯一入口（REST），所有组件通过它交互，唯一读写 etcd 的组件。
- **etcd**：分布式 KV（Raft 强一致），存集群全部状态，**唯一事实源**。
- **Scheduler**：为新 Pod 选合适的 Node（过滤 + 打分）。
- **Controller Manager**：一堆控制器（Deployment、ReplicaSet、Node 等），通过**控制循环**让实际状态不断逼近期望状态（声明式核心）。

**Node / 数据面**：

- **kubelet**：管理本节点 Pod 生命周期，对接 CRI，上报状态。
- **kube-proxy**：维护网络规则（iptables/IPVS），实现 Service 负载均衡。
- **容器运行时**：containerd / CRI-O。

> 声明式精髓：你只声明"期望状态"（YAML），Controller 持续调和（reconcile）让现实达到期望，故障后能自愈。

### 12. 核心对象 🔥

| 对象 | 作用 |
| --- | --- |
| **Pod** | 最小调度单元，1+ 个共享网络/存储的容器；一般不直接建，由控制器管理 |
| **ReplicaSet** | 保证 Pod 副本数（通常由 Deployment 管理） |
| **Deployment** | 无状态应用，管副本数 + 滚动更新 + 回滚 |
| **StatefulSet** | 有状态应用（DB/中间件），稳定网络标识 + 独立持久存储 + 有序部署 |
| **DaemonSet** | 每节点跑一个（日志采集、监控、网络插件） |
| **Job / CronJob** | 一次性任务 / 定时任务 |
| **Service** | 一组 Pod 的稳定访问入口（虚拟 IP + 负载均衡 + 服务发现） |
| **Ingress** | 七层路由，把外部 HTTP(S) 流量按域名/路径路由到 Service |
| **ConfigMap / Secret** | 非敏感配置 / 敏感信息（Secret 仅 Base64，需配 RBAC + 加密） |
| **PV / PVC / StorageClass** | 持久卷 / 卷声明 / 动态供给 |
| **Namespace** | 逻辑隔离（环境/团队），配合 ResourceQuota |

### 13. 资源管理：requests/limits 与 QoS 🔥（高频）

每个容器可设 `requests`（**调度依据**，预留资源）和 `limits`（**运行上限**）：

- CPU 超 limit 被**限流（throttle）**；内存超 limit 被 **OOM Kill**。
- 三种 **QoS 等级**（影响节点资源紧张时的驱逐顺序）：
  - **Guaranteed**：requests == limits，最稳，最后被驱逐。
  - **Burstable**：设了 requests < limits。
  - **BestEffort**：都没设，最先被驱逐。
- 实战：**内存 requests=limits（避免被驱逐）**，CPU limits 谨慎设（设太低反而 throttle）。配合 JVM 的 `MaxRAMPercentage`（见第 5 题）。

### 14. 调度：亲和性、污点与容忍 ⭐

- **nodeSelector / Node 亲和性**：把 Pod 调度到带特定标签的节点（如 GPU 节点）。
- **Pod 亲和/反亲和**：让相关 Pod 靠近（同机房降延迟）或分散（同一应用副本分到不同节点，提高可用性）。
- **污点（Taint）与容忍（Toleration）**：给节点打污点"排斥"Pod，只有声明容忍的 Pod 才能调度上去（如专用节点、master 节点）。

### 15. 网络模型 ⭐

- **Pod 网络**：每个 Pod 一个独立 IP，集群内 Pod 间可直接互通（扁平网络），由 **CNI 插件**（Calico / Flannel / Cilium）实现。
- **Service 实现**：`kube-proxy` 用 **iptables**（规则多时性能下降）或 **IPVS**（大规模更优）做 ClusterIP 的负载均衡。
- **DNS**：CoreDNS 提供服务发现，`服务名.命名空间.svc.cluster.local`。
- **Service 类型**：ClusterIP（内部，默认）、NodePort（节点端口暴露）、LoadBalancer（云 LB）、Ingress（七层网关，做域名/路径路由，生产入口首选）。

### 16. 探针（健康检查）🔥

- **livenessProbe（存活）**：失败 → **重启**容器（救活假死进程）。
- **readinessProbe（就绪）**：失败 → 从 Service 端点**摘除**（不接流量），就绪后再加入（保证只把流量给能服务的 Pod）。
- **startupProbe（启动）**：保护慢启动应用（Spring Boot 启动慢），启动期间不被 liveness 误杀。

> 配合**优雅停机**：K8s 发 `SIGTERM` → 应用停止接新请求、处理完存量（Spring Boot graceful shutdown）→ `preStop` 钩子兜底 → 超 `terminationGracePeriodSeconds` 才 `SIGKILL`。保证发布/缩容不丢请求。

### 17. 弹性伸缩 ⭐

- **HPA**：按 CPU/内存/自定义指标（QPS、队列长度）自动增减 **Pod 数**（水平）。
- **VPA**：自动调单 Pod 的 requests/limits（垂直）。
- **Cluster Autoscaler**：Pod 因资源不足挂起时，自动加 **Node**。
- 进阶：**KEDA** 支持基于事件（Kafka 堆积、MQ 长度）伸缩，可缩容到 0。

### 18. 配置与存储 ⭐

- **ConfigMap/Secret**：以环境变量或挂载文件注入容器，变更可不重建镜像。Secret 默认只是 Base64，需开 etcd 加密 + RBAC。
- **存储**：`emptyDir`（临时）、`hostPath`（节点路径，慎用）、**PV/PVC + StorageClass**（动态供给云盘/NFS/Ceph），有状态应用用 StatefulSet + PVC。

## 三、发布策略与编排工具

### 19. 发布策略 🔥

| 策略 | 做法 | 特点 |
| --- | --- | --- |
| **滚动更新（默认）** | 逐批替换旧 Pod（`maxSurge`/`maxUnavailable` 控速） | 平滑、无停机、回滚稍慢 |
| **蓝绿** | 新旧两套环境，切流量一次性切换 | 切换快、可秒回滚，但要双倍资源 |
| **金丝雀/灰度** | 先放小比例流量给新版本，验证后逐步放量 | 风险小，需流量治理（Ingress/Service Mesh 按权重/标签） |
| **重建（Recreate）** | 先停全部旧的再起新的 | 有停机，仅适合非在线 |

回滚：`kubectl rollout undo deployment/xxx`，Deployment 保留历史 ReplicaSet。

### 20. Helm 与 Operator/CRD ⭐

- **Helm**：K8s 的"包管理器"，用 **Chart（模板 + values）** 管理一组资源 YAML，支持参数化、版本管理、一键安装/升级/回滚。解决"几十个 YAML 难维护、多环境差异"。
- **CRD（自定义资源定义）+ Operator**：扩展 K8s API，把**运维知识代码化**。Operator 用自定义控制器管理有状态中间件（如自动部署/扩容/备份 MySQL、Redis、Kafka），把 DBA 的操作自动化。

## 四、CI/CD 与 DevOps 🔥

### 21. CI/CD 是什么

- **CI（持续集成）**：开发频繁合并代码，自动触发**构建 → 单测 → 代码扫描 → 打镜像**，尽早发现问题。
- **CD（持续交付/部署）**：自动把通过的制品**发布到测试/预发/生产**（持续交付到预发需人工点一下上线；持续部署全自动到生产）。
- 价值：**快速迭代、降低发布风险、可重复、可回滚**。

### 22. 一条典型流水线 🔥

```
代码提交(Git) → 触发 CI
  → 编译构建 → 单元测试 → 代码质量扫描(SonarQube) → 安全扫描(Trivy/SCA)
  → 构建镜像 → 推送镜像仓库(Harbor)
  → CD：部署到测试环境 → 自动化测试 → 人工审批
  → 灰度发布生产 → 监控验证 → 全量 / 自动回滚
```

GitLab CI 示例（`.gitlab-ci.yml` 片段）：

```yaml
stages: [build, test, image, deploy]

build:
  stage: build
  script: mvn -B clean package -DskipTests

test:
  stage: test
  script: mvn -B test

image:
  stage: image
  script:
    - docker build -t harbor.xxx.com/app:$CI_COMMIT_SHORT_SHA .
    - docker push harbor.xxx.com/app:$CI_COMMIT_SHORT_SHA

deploy:
  stage: deploy
  script:
    - kubectl set image deployment/app app=harbor.xxx.com/app:$CI_COMMIT_SHORT_SHA
  only: [main]
```

常用工具：**Jenkins / GitLab CI / GitHub Actions / Tekton**（CI）；**Argo CD / Flux**（CD/GitOps）。

### 23. 镜像构建工具 ⭐（Java 相关）

- **Docker build**：最通用，需 Docker daemon。
- **Jib（Google）**：**Maven/Gradle 插件，无需 Dockerfile、无需 Docker daemon** 即可构建并推送 Java 镜像，自动分层（依赖/资源/类分开，改代码只推变化层），CI 友好，**Java 项目强烈推荐**。
- **Kaniko**：在容器/K8s 内**无 daemon、无特权**构建镜像，适合在 CI 集群里安全构建。
- **Cloud Native Buildpacks**（`pack` / Spring Boot `bootBuildImage`）：无需写 Dockerfile，自动探测语言生成优化镜像。

### 24. GitOps（CD 的现代范式）🔥

以 **Git 仓库作为唯一事实源**，用声明式 YAML 描述集群期望状态，由 **Argo CD / Flux** 持续监听 Git 并**自动同步**到集群，保证集群实际状态 == Git 声明。

- 优点：**所有变更可审计（Git 历史）、可回滚（git revert）、环境一致、自动纠偏（人工改了集群会被拉回 Git 状态）**。
- 流程：CI 构建镜像并更新部署仓库的镜像 tag → Argo CD 检测到 Git 变更 → 自动 apply 到集群。

### 25. DevOps 与 DevSecOps ⭐

- **DevOps**：打通开发与运维，靠自动化（CI/CD）+ 文化协作，实现快速、可靠交付。
- **DevSecOps**：把**安全左移**到流水线（依赖漏洞扫描 SCA、镜像扫描、密钥检测、合规卡点），让安全成为流水线的自动环节而非事后补救。

## 五、Service Mesh

### 26. Service Mesh（服务网格）⭐

把服务治理（熔断、限流、重试、链路追踪、mTLS 加密、灰度）从业务代码**下沉到 Sidecar 代理**（如 **Istio + Envoy**），业务无感知、跨语言统一治理。

- **数据面**：每个 Pod 注入 Sidecar（Envoy），拦截所有进出流量。
- **控制面**：Istiod 统一下发路由/策略配置。
- **对比 Spring Cloud**：治理能力从 **SDK（侵入业务、绑 Java）** 转移到**基础设施层（无侵入、跨语言）**；代价是多一跳代理的性能损耗和运维复杂度。
- 与 [07-SpringCloud](./07-SpringCloud.md) 的演进关系：库治理 → 平台治理。
