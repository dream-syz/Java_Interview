# 26 · 云原生（Docker / Kubernetes）

> 覆盖：容器原理、Docker 核心、镜像分层、Kubernetes 核心对象与架构、调度与扩缩容、Service/Ingress、滚动发布、Service Mesh、CI/CD。现代后端的基础设施常识。

---

## 一、Docker

### 1. 容器与虚拟机的区别 🔥

| | 容器 | 虚拟机 |
| --- | --- | --- |
| 隔离级别 | 进程级（共享宿主内核） | 操作系统级（独立 OS） |
| 资源开销 | 小（MB 级，秒级启动） | 大（GB 级，分钟级启动） |
| 性能 | 接近原生 | 有虚拟化损耗 |
| 隔离性 | 较弱 | 强 |

### 2. 容器隔离的底层原理 ⭐

- **Namespace（命名空间）**：隔离**视图**（PID、网络、挂载、用户、主机名等），让容器以为自己独占系统。
- **Cgroups（控制组）**：限制**资源**（CPU、内存、IO 配额）。
- **UnionFS（联合文件系统）**：镜像分层与写时复制。

> 一句话：**Namespace 管隔离，Cgroups 管限制，UnionFS 管镜像**。容器本质是宿主机上的一个被隔离和限制的进程。

### 3. 镜像分层与写时复制 🔥

- 镜像由多个**只读层**叠加（每条 Dockerfile 指令一层），层可被多镜像**共享复用**（省空间、加速构建/拉取）。
- 容器运行时在镜像顶部加一个**可写层**，修改文件时**写时复制（CoW）** 到可写层，不影响镜像。

### 4. Dockerfile 优化 ⭐

- **多阶段构建**：编译阶段用大镜像，运行阶段只拷贝产物到小镜像（减小体积）。
- 合理利用**层缓存**：变动少的指令放前面（如先 `COPY pom.xml` 再下依赖，最后 `COPY` 源码）。
- 用精简基础镜像（alpine / distroless）、合并 RUN 减少层数、`.dockerignore`。

### 5. 常用命令 ⭐

`docker build/run/ps/images/exec/logs/rm/rmi`、`docker-compose up`（多容器编排）。

## 二、Kubernetes（K8s）

### 6. 为什么需要 K8s 🔥

Docker 解决了"单个容器怎么跑"，K8s 解决"**大规模容器怎么编排管理**"：自动调度、自愈（故障重启）、弹性伸缩、滚动发布、服务发现与负载均衡、配置与密钥管理。

### 7. 架构（控制面 + 数据面）🔥

**Master（控制面）**：
- **API Server**：所有操作的统一入口（REST），唯一与 etcd 交互。
- **etcd**：分布式 KV，存集群所有状态（唯一数据源）。
- **Scheduler**：决定 Pod 调度到哪个 Node。
- **Controller Manager**：各种控制器，保证实际状态向期望状态收敛。

**Node（数据面）**：
- **kubelet**：管理本节点 Pod 生命周期，上报状态。
- **kube-proxy**：维护网络规则，实现 Service 的负载均衡。
- **容器运行时**：containerd / CRI-O（Docker 已弃用 dockershim）。

### 8. 核心对象 🔥

| 对象 | 作用 |
| --- | --- |
| **Pod** | 最小调度单元，一个或多个共享网络/存储的容器 |
| **Deployment** | 管理无状态应用，控制 Pod 副本数、滚动更新、回滚 |
| **StatefulSet** | 有状态应用（DB），稳定网络标识和存储 |
| **DaemonSet** | 每个节点跑一个（日志采集、监控 agent） |
| **Service** | 一组 Pod 的稳定访问入口（虚拟 IP + 负载均衡） |
| **Ingress** | 七层路由，把外部 HTTP 流量路由到 Service |
| **ConfigMap / Secret** | 配置 / 密钥 |
| **PV / PVC** | 持久化存储 |
| **Namespace** | 逻辑隔离（环境/团队） |

### 9. Service 的类型 ⭐

- **ClusterIP**（默认）：集群内部访问。
- **NodePort**：每个节点开端口，外部可访问。
- **LoadBalancer**：对接云厂商负载均衡器。
- **Ingress**：七层入口，做域名/路径路由（比 NodePort 优雅）。

### 10. 健康检查（探针）🔥

- **livenessProbe（存活）**：失败则**重启**容器。
- **readinessProbe（就绪）**：失败则从 Service **摘除**（不接流量），就绪后再加入。
- **startupProbe（启动）**：慢启动应用，启动期间不被 liveness 误杀。

> 配合优雅停机（`preStop` + `SIGTERM` 处理）保证发布无损。

### 11. 弹性伸缩 ⭐

- **HPA（Horizontal Pod Autoscaler）**：按 CPU/内存/自定义指标自动增减 Pod 数。
- **VPA**：调整单 Pod 的资源请求。
- **Cluster Autoscaler**：自动增减 Node。

### 12. 滚动发布与回滚 🔥

Deployment 默认**滚动更新**（逐批替换 Pod，`maxSurge`/`maxUnavailable` 控制节奏），失败可 `kubectl rollout undo` 回滚。其它策略：蓝绿、金丝雀（配合 Ingress/Service Mesh 按流量比例）。

## 三、Service Mesh 与 CI/CD

### 13. Service Mesh（服务网格）⭐

把服务治理（熔断、限流、链路、mTLS、灰度）从业务代码**下沉到 Sidecar 代理**（如 Istio + Envoy），业务无感知、跨语言。

- **数据面**：Sidecar（Envoy）拦截所有进出流量。
- **控制面**：Istiod 下发配置。
- 对比 Spring Cloud：治理能力从 SDK（侵入、绑语言）转移到基础设施层（无侵入、跨语言），代价是运维复杂度和性能损耗。

### 14. CI/CD ⭐

- **CI（持续集成）**：代码提交 → 自动构建 → 测试 → 打镜像（Jenkins / GitLab CI / GitHub Actions）。
- **CD（持续部署）**：自动发布到环境（ArgoCD / GitOps）。
- **GitOps**：以 Git 为唯一事实源，声明式描述期望状态，自动同步到集群。
