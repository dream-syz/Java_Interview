# 10 · Redis

> 覆盖：为什么快、数据类型与底层编码、线程模型与多线程、持久化（RDB/AOF）、过期与淘汰、缓存三大问题（穿透/击穿/雪崩）、双写一致性、分布式锁、集群与高可用。

---

## 一、基础与线程模型

### 1. Redis 为什么快 🔥

1. **纯内存**操作。
2. **单线程**（命令处理）避免锁竞争与上下文切换。
3. **IO 多路复用**（epoll）的事件驱动模型，单线程高效处理大量连接。
4. 高效的**数据结构**（SDS、跳表、压缩列表等）。
5. 自研协议 RESP，简单高效。

### 2. Redis 是单线程还是多线程 🔥

- **核心命令处理始终是单线程**（保证原子性、避免并发问题）。
- Redis **4.0** 引入后台线程处理异步删除（`UNLINK`、异步刷盘）。
- Redis **6.0** 引入**多线程 IO**：用多线程处理网络读写与协议解析，**命令执行仍单线程**。目的是分摊网络 IO 开销（瓶颈常在网络而非 CPU）。

### 3. Redis 与 Memcached 的区别 ⭐

Redis 支持丰富数据结构、持久化、主从/集群、发布订阅、Lua、事务；Memcached 只有简单 KV、多线程、无持久化。**Redis 全面胜出，新项目首选 Redis**。

## 二、数据类型与底层

### 4. 五种基本类型 + 底层编码 🔥

| 类型 | 用途 | 底层编码 |
| --- | --- | --- |
| String | 缓存、计数、分布式锁 | int / embstr（≤44字节）/ raw（SDS） |
| Hash | 对象属性 | listpack（小）/ hashtable |
| List | 队列、栈、时间线 | quicklist（listpack 链表） |
| Set | 去重、共同好友 | intset（纯整数小集合）/ hashtable |
| ZSet | 排行榜、延迟队列 | listpack（小）/ **跳表 skiplist + hashtable** |

进阶类型：Bitmap（签到/布隆）、HyperLogLog（基数统计，UV）、GEO（地理位置）、Stream（消息队列）。

### 5. ZSet 为什么用跳表而不用红黑树 ⭐

跳表实现简单、范围查询（`ZRANGE`）效率高（叶子有序链表，无需中序遍历）、并发友好，且作者认为实现和维护成本更低。复杂度同为 O(log n)。

### 6. SDS（简单动态字符串）⭐

Redis 的字符串不用 C 原生字符串，而用 SDS：记录 `len`/`alloc`，**O(1) 取长度**、**二进制安全**、**杜绝缓冲区溢出**、空间预分配与惰性释放减少内存重分配。

## 三、持久化

### 7. RDB 与 AOF 🔥

| | RDB | AOF |
| --- | --- | --- |
| 形式 | 某时刻的**内存快照**（二进制） | 记录**写命令**（追加日志） |
| 触发 | `save`(阻塞)/`bgsave`(fork 子进程) / 定时 | 每条写命令追加 |
| 优点 | 文件小、恢复快、适合备份 | 数据更安全（丢失少） |
| 缺点 | 可能丢失最后一次快照后的数据 | 文件大、恢复慢 |
| 刷盘 | — | `appendfsync`：always/**everysec**(默认)/no |

- **AOF 重写（rewrite）**：fork 子进程，把当前数据集压缩成最小命令集，控制文件膨胀。
- **混合持久化（4.0+，推荐）**：AOF 重写时前半段用 RDB 格式，后半段用 AOF 增量命令，兼顾恢复速度与数据安全。
- 取舍：可丢少量数据用 RDB；要求高可靠用 AOF everysec 或混合持久化。

## 四、过期与淘汰

### 8. 过期删除策略 🔥

Redis 采用**惰性删除 + 定期删除**组合：

- **惰性删除**：访问 key 时才检查是否过期，过期则删（节省 CPU，但可能留垃圾）。
- **定期删除**：周期性随机抽样部分 key 删除过期的（限制时长，避免阻塞）。
- 不会主动扫描全部（太耗 CPU）。残留的过期 key 由内存淘汰兜底。

### 9. 内存淘汰策略 🔥

`maxmemory` 满后按 `maxmemory-policy` 淘汰：

- `noeviction`（默认，不淘汰，写报错）。
- `allkeys-lru` / `volatile-lru`（最近最少使用）。
- `allkeys-lfu` / `volatile-lfu`（最不经常使用，4.0+，更精准）。
- `allkeys-random` / `volatile-random`。
- `volatile-ttl`（优先淘汰快过期的）。

> Redis 的 LRU/LFU 是**近似算法**（抽样），非严格全局，省内存与 CPU。

## 五、缓存三大问题（高频）

### 10. 缓存穿透 🔥

查询**不存在**的数据，缓存不命中每次都打到 DB（常被恶意攻击）。

解决：
- **缓存空值**（`null` 设短 TTL）。
- **布隆过滤器**：拦截一定不存在的 key（有小概率误判但不会漏判存在）。
- 接口校验、限流。

### 11. 缓存击穿 🔥

**单个热点 key 过期**瞬间，大量并发同时打到 DB。

解决：
- **互斥锁/分布式锁**：只让一个线程查 DB 重建缓存，其余等待。
- **热点 key 逻辑过期**（不设物理过期，后台异步更新）。

### 12. 缓存雪崩 🔥

**大量 key 同时过期**或 **Redis 宕机**，请求集中打到 DB。

解决：
- 过期时间加**随机抖动**，避免同时失效。
- 多级缓存、**Redis 高可用集群**。
- 熔断限流、服务降级保护 DB。

### 13. 缓存与数据库双写一致性 🔥🔥

推荐 **Cache Aside（旁路缓存）+ 先更新 DB 再删除缓存**：

- 读：先读缓存，未命中读 DB 并回填缓存。
- 写：**先更新数据库，再删除缓存**（而非更新缓存）。
- 为什么删除而非更新：避免并发写导致缓存脏数据，且缓存懒加载更省。
- 仍有极端不一致窗口（读请求在删除前回填了旧值）→ **延迟双删**（删缓存 → 更新 DB → 延迟再删一次）。
- 强一致场景：用 **binlog 订阅（Canal）异步删缓存**，或接受最终一致。
- 兜底：给缓存设置合理 TTL，保证最终一致。

## 六、分布式锁（高频）

### 14. 用 Redis 实现分布式锁 🔥🔥

```redis
SET lock_key unique_value NX PX 30000   # 原子：不存在才设置 + 过期时间
```

- `NX` 保证互斥，`PX` 设过期防死锁，`unique_value` 标识持有者。
- **释放锁**必须用 **Lua 脚本**保证“判断是自己的锁 + 删除”原子：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
  return redis.call('del', KEYS[1])
else return 0 end
```

存在的问题与进阶：
- **锁过期但业务没执行完** → 别人拿到锁。解决：**看门狗（watchdog）自动续期**（Redisson 实现）。
- 主从切换可能丢锁（主写入未同步从就宕机）→ **RedLock** 算法（多数派），但有争议，对强一致要求高用 **Zookeeper/etcd**。
- 生产推荐直接用 **Redisson**（封装了可重入、续期、公平锁等）。

## 七、其他与集群

### 15. Redis 事务 ⭐

`MULTI`/`EXEC`/`DISCARD`/`WATCH`。特点：命令入队、`EXEC` 顺序执行，**不支持回滚**（命令执行期出错不影响其它命令），不是真正的 ACID。复杂原子逻辑用 **Lua 脚本**（单线程执行天然原子）。

### 16. 主从复制与哨兵 🔥

- **主从复制**：从节点复制主节点数据，读写分离、数据冗余。全量同步（RDB）+ 增量同步（repl backlog）。
- **哨兵（Sentinel）**：监控主从，主节点宕机时**自动故障转移**（选新主、通知客户端），实现高可用。

### 17. Redis Cluster（集群分片）🔥

- **16384 个哈希槽**，`CRC16(key) % 16384` 决定 key 落到哪个槽，槽分布到各主节点，实现数据分片与水平扩展。
- 每个主节点配从节点做高可用；节点间用 Gossip 协议通信。
- 不支持跨槽的多 key 操作（可用 `{hashtag}` 把相关 key 固定到同槽）。
- 集群不可用场景：超过半数主节点故障，或某主节点及其所有从节点都挂（该槽无人服务）。

### 18. 什么情况会导致 Redis 阻塞 ⭐

大 key 删除/查询、`keys *`、`save` 全量持久化、AOF `always` 刷盘、fork 子进程时大内存复制页表、慢 Lua、swap、网络/CPU 打满。**避免大 key、用 `scan` 替代 `keys`、用 `UNLINK` 异步删除**。

### 19. 什么是 bigkey，影响与处理 ⭐

bigkey：value 很大（大 String / 元素极多的集合）。影响：内存不均、阻塞（删除/迁移耗时）、网络拥塞。处理：拆分、用合适结构、`UNLINK` 异步删除、`scan` + 渐进删除。

### 20. pipeline 的作用 ⭐

把多条命令打包一次性发送，**减少 RTT 网络往返**，大幅提升批量操作吞吐。注意它不是事务（无原子性保证），且单批不宜过大。

### 21. 多级缓存架构 ⭐（进阶）

`本地缓存（Caffeine）+ 分布式缓存（Redis）` 组合：

- **本地缓存（Caffeine）**：进程内、纳秒级、无网络开销，挡住**超级热点 key**（避免热点 key 把单个 Redis 节点打爆）。基于 W-TinyLFU 淘汰算法，命中率高于 Guava Cache。
- **Redis**：跨节点共享、容量大，挡住 DB。
- **一致性**：本地缓存有各节点不一致问题，靠**短 TTL + 变更时 MQ/发布订阅广播失效**解决。
- 读链路：本地缓存 → Redis → DB（逐层回填）。详见 [24-高并发高可用](./24-高并发高可用.md)。

### 22. 热点 key 问题 ⭐

- **发现**：监控 Redis 单 key QPS、用 `redis-cli --hotkeys`、客户端埋点统计。
- **解决**：本地缓存兜住、**key 打散**（`key#1`~`key#N` 分散到多个节点，读时随机选一个）、只读副本扩容。

### 23. Redisson 看门狗（watchdog）原理 ⭐

加锁不指定过期时间时，Redisson 默认 30s 过期，并启动一个**后台定时任务**每 `10s`（1/3 过期时间）检查锁是否还被持有，是则**续期到 30s**，防止业务没执行完锁就过期。释放锁时取消续期。指定了过期时间则不启用看门狗。
