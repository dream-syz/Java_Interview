# 11 · 消息队列 MQ

> 覆盖：为什么用 MQ、主流选型、可靠投递（不丢消息）、消息不重复（幂等）、顺序消费、消息积压、重复消费、事务消息、如何设计一个 MQ。以 Kafka / RocketMQ / RabbitMQ 为主。

---

## 一、为什么用 MQ

### 1. MQ 的三大核心作用 🔥

- **解耦**：上下游通过消息通信，互不依赖。
- **异步**：非核心流程异步化，缩短主流程响应时间。
- **削峰填谷**：流量高峰时消息堆积在 MQ，消费者按自身能力消费，保护下游。

### 2. MQ 的缺点 ⭐

- **系统可用性降低**：MQ 挂了影响整个链路（需高可用）。
- **复杂度提升**：要处理重复消费、顺序、一致性、积压。
- **一致性问题**：异步可能导致数据不一致。

### 3. 主流 MQ 选型 🔥

| 维度 | RabbitMQ | Kafka | RocketMQ |
| --- | --- | --- | --- |
| 语言 | Erlang | Scala/Java | Java |
| 吞吐 | 万级 | **百万级** | 十万~百万级 |
| 延迟 | 微秒级（最低） | 毫秒级 | 毫秒级 |
| 功能 | 灵活路由 | 流处理、日志 | 事务/顺序/延迟消息齐全 |
| 适用 | 业务解耦、低延迟 | **大数据、日志、流** | **金融级业务、电商** |

> 选型经验：日志/大数据流 → Kafka；电商/金融业务且要事务/顺序/延迟消息 → RocketMQ；中小业务、复杂路由 → RabbitMQ。

## 二、可靠性（不丢消息）🔥🔥

消息丢失可能发生在三个环节，要逐段保证：

### 4. 生产者不丢

- 同步发送 + 确认机制：Kafka `acks=all`（所有 ISR 副本确认）+ 重试；RabbitMQ **confirm 机制**；RocketMQ 同步发送返回 SEND_OK。
- 发送失败重试 + 本地消息表兜底。

### 5. MQ 自身不丢

- **持久化 + 多副本**：Kafka `replication.factor>=3`、`min.insync.replicas>=2`；RabbitMQ 队列和消息持久化 + 镜像/Quorum 队列；RocketMQ 同步刷盘 + 主从同步复制。
- 关闭可能丢数据的优化（如 Kafka `unclean.leader.election`）。

### 6. 消费者不丢

- **手动 ACK**：消费**处理成功后**再确认，而非拉到就确认（自动 ACK 在处理失败时会丢）。
- 消费失败重试 + **死信队列（DLQ）** 兜底。

## 三、重复消费与幂等 🔥

### 7. 如何保证不重复消费（幂等）

MQ 通常保证 **at-least-once（至少一次）**，无法天然 exactly-once，所以**消费端必须幂等**：

- 业务唯一键 + 数据库唯一索引（重复插入失败）。
- 用 Redis/DB 记录已处理的消息 ID（去重表）。
- 状态机：只允许特定状态流转。
- 乐观锁（version）。

> 详见 [15-分布式](./15-分布式.md) 幂等设计。

## 四、顺序消费 🔥

### 8. 如何保证消息顺序

- **全局顺序**：单分区/单队列 + 单消费者（牺牲吞吐，少用）。
- **局部顺序（推荐）**：把需要有序的消息（如同一订单）按 key **路由到同一分区/队列**，分区内有序，消费者单线程消费该分区。
  - Kafka：同一 key → 同一 partition。
  - RocketMQ：`MessageQueueSelector` 按订单 ID 选队列，顺序消费用 `MessageListenerOrderly`。
- 注意消费端多线程会打乱顺序，需保证单分区单线程处理。

## 五、积压与延时 🔥

### 9. 几百万消息积压几小时怎么解决

- 临时**扩容消费者**：但受分区数限制（Kafka 消费者数 ≤ 分区数）。
- 紧急方案：写一个临时分发程序，把积压消息**快速转发到一个新建的、分区数更多的临时 topic**，再用大量临时消费者并行处理。
- 事后：排查消费慢的原因（下游慢、消费逻辑重），优化消费性能，提前做好容量规划与监控告警。

### 10. 消息过期失效 / 队列满了

- 设置了 TTL 的消息过期会被丢弃 → 高峰期临时关闭 TTL，事后**手动写程序补偿**重发丢失的消息。
- 队列满 → 触发拒绝/死信，需扩容 + 限流生产者。

### 11. 延迟消息 ⭐

- RocketMQ：内置延迟级别（或 5.0 任意时间延迟）。
- RabbitMQ：TTL + 死信队列，或 `delayed-message-exchange` 插件。
- Redis：ZSet（score 为到期时间）轮询，或 Redisson 延迟队列。

## 六、事务消息与设计

### 12. RocketMQ 事务消息 🔥

解决“本地事务 + 发消息”的原子性（典型：下单扣库存）：

1. 发送**半消息（half message）**，对消费者不可见。
2. 执行本地事务。
3. 本地事务成功 → **commit**（消息可见）；失败 → rollback（丢弃半消息）。
4. 若 Broker 未收到二次确认，**回查**生产者本地事务状态，决定提交还是回滚。

是分布式事务“**最终一致**”的经典实现，类似本地消息表。

### 13. 让你设计一个 MQ，怎么设计 ⭐（架构题）

- **存储**：顺序写磁盘 + 分段日志（参考 Kafka 的分区 + 顺序追加 + 零拷贝），保证高吞吐。
- **高可用**：分区多副本 + Leader/Follower + 故障自动切换。
- **可扩展**：Topic 分区化，水平扩展 Broker。
- **可靠性**：刷盘策略、副本同步、生产/消费确认机制。
- **消费模型**：消费组 + offset 管理，支持广播/集群消费。
- 配套：消息索引、重试、死信、延迟、监控。
