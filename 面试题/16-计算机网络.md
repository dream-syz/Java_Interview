## 二、TCP / UDP

### 2. TCP 与 UDP 的区别 🔥

| | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠（确认、重传、有序） | 不可靠 |
| 速度 | 较慢 | 快 |
| 头部 | 20 字节起 | 8 字节 |
| 场景 | HTTP、文件传输 | 视频/直播、DNS、游戏 |

### 3. 三次握手 🔥🔥

```
Client          Server
  | --- SYN(seq=x) ----------> |   client: SYN_SENT
  | <-- SYN(seq=y)ACK(x+1) --- |   server: SYN_RCVD
  | --- ACK(y+1) ------------> |   established
```

**为什么三次而不是两次**：两次握手无法确认**客户端的接收能力**和防止**历史失效连接请求**突然到达造成服务端建立无效连接。三次才能让双方都确认彼此的收发能力。

### 4. 四次挥手 🔥🔥

```
Client                       Server
  | --- FIN ------------------> |   client: FIN_WAIT_1
  | <-- ACK ------------------- |   server: CLOSE_WAIT
  | <-- FIN ------------------- |   server: LAST_ACK
  | --- ACK ------------------> |   client: TIME_WAIT → CLOSED
```

**为什么四次**：TCP 全双工，关闭需双向各自关闭。服务端收到 FIN 后可能还有数据要发，所以 ACK 和 FIN **分两次**发。

**TIME_WAIT（2MSL）的作用**：① 保证最后一个 ACK 能到达（丢失可重传）；② 让本次连接的旧报文在网络中消散，避免影响新连接。

### 5. TCP 如何保证可靠传输 🔥

序列号 + 确认应答（ACK）、超时重传、**滑动窗口**（流量控制）、**拥塞控制**（慢启动、拥塞避免、快重传、快恢复）、校验和、有序重组。

### 6. 流量控制 vs 拥塞控制 ⭐

- **流量控制**：滑动窗口，防止发送方太快淹没**接收方**。
- **拥塞控制**：防止过多数据注入**网络**，调节拥塞窗口 cwnd（慢启动指数增长 → 拥塞避免线性增长 → 丢包后快重传/快恢复）。

### 7. 粘包/拆包 ⭐

TCP 是字节流，没有消息边界 → 粘包/拆包。解决：固定长度、分隔符、**消息头带长度字段**（最常用，Netty 的 `LengthFieldBasedFrameDecoder`）。

## 三、HTTP / HTTPS

### 8. 常见 HTTP 方法与幂等 ⭐

GET（查，幂等）、POST（增，不幂等）、PUT（全量更新，幂等）、DELETE（删，幂等）、PATCH（部分更新）、HEAD、OPTIONS。

### 9. 常见状态码 🔥

- 2xx：200 OK、201 Created、204 No Content。
- 3xx：301 永久重定向、302 临时、304 Not Modified（协商缓存命中）。
- 4xx：400 参数错、401 未认证、403 禁止、404 不存在、429 限流。
- 5xx：500 服务器错、502 网关错、503 不可用、504 网关超时。

### 10. GET 与 POST 的区别 ⭐

GET 参数在 URL（有长度限制、会被缓存/记录、幂等），POST 参数在 body（更适合大量/敏感数据、不幂等）。本质都是 TCP 上的 HTTP，安全性都需 HTTPS 保障。

### 11. HTTP 与 HTTPS 🔥

- HTTP 明文传输，不安全；HTTPS = **HTTP + TLS/SSL**，加密 + 身份认证 + 完整性校验，端口 443。
- **TLS 握手（混合加密）**：
  1. 客户端发起，服务端返回**证书（含公钥）**。
  2. 客户端验证证书（CA 链），生成随机数（会话密钥）用**公钥加密**发给服务端（**非对称加密**传递密钥）。
  3. 之后用协商出的**对称密钥**加密通信（**对称加密**传数据，性能高）。
- **对称加密**（AES，快，但密钥分发难）+ **非对称加密**（RSA/ECC，安全分发密钥，慢）结合。
- 数字证书 + CA 解决**中间人攻击**和身份信任问题。

### 12. HTTP/1.1、HTTP/2、HTTP/3 🔥

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 连接 | 持久连接、串行（队头阻塞） | **多路复用**（一连接多流） | 多路复用 |
| 传输 | 文本 | **二进制分帧** | 二进制 |
| 头部 | 不压缩 | **HPACK 头压缩** | QPACK |
| 底层 | TCP | TCP（仍有 TCP 队头阻塞） | **QUIC（基于 UDP）**，解决 TCP 队头阻塞 |
| 推送 | 无 | Server Push（已少用） | — |

### 13. Cookie 与 Session ⭐

- **Cookie**：存在客户端，随请求自动携带。
- **Session**：存在服务端，用 Cookie 里的 sessionId 关联。
- 分布式下 Session 共享问题 → Redis 集中存储，或用无状态 **JWT Token**。

## 四、综合

### 14. 输入 URL 到页面展示的全过程 🔥（经典综合题）

1. **DNS 解析**：域名 → IP（浏览器缓存 → 系统 → 路由器 → DNS 服务器递归查询）。
2. **建立 TCP 连接**（三次握手），HTTPS 还要 TLS 握手。
3. **发送 HTTP 请求**。
4. 服务器处理并返回响应（可能经 CDN、Nginx、网关、微服务）。
5. 浏览器**解析渲染**（构建 DOM/CSSOM → 渲染树 → 布局 → 绘制），加载 JS/CSS/图片。
6. **关闭连接**（四次挥手，或 keep-alive 复用）。

### 15. 序列化协议对比 ⭐（RPC 相关）

| 协议 | 特点 | 场景 |
| --- | --- | --- |
| JSON | 文本、可读、跨语言 | HTTP API（通用） |
| **Protobuf** | 二进制、紧凑、快、跨语言、需 IDL | gRPC、高性能 RPC |
| Thrift | 二进制 + RPC 框架一体 | 跨语言服务 |
| Hessian | 二进制、Java 友好 | Dubbo 默认 |
| Java 原生 | 慢、有安全漏洞、不跨语言 | 不推荐 |

衡量维度：体积、编解码速度、跨语言、可读性、是否需要 IDL。高性能 RPC 选 Protobuf。

### 16. WebSocket ⭐

基于 TCP 的**全双工**协议，通过 HTTP `Upgrade` 握手升级而来，建立后服务端可主动推送，适合实时通信（IM、推送、协同），见 [23-系统设计](./23-系统设计.md) IM 部分。

- TIME_WAIT 有什么用、为什么 2MSL？→ 第 4 题。
- HTTPS 怎么做到既安全又快？→ 非对称传密钥 + 对称传数据（第 11 题）。
- HTTP/2 多路复用解决了什么？还有什么没解决？→ 应用层队头阻塞解决，TCP 层未解决（HTTP/3 解决）（第 12 题）。
