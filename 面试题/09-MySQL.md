## 一、引擎与范式

### 1. 三大范式 ⭐

- 1NF：列不可再分（原子性）。
- 2NF：非主属性完全依赖主键（消除部分依赖）。
- 3NF：非主属性不依赖其他非主属性（消除传递依赖）。
- 实践中常**反范式**（冗余字段）以减少 join、提升读性能，按业务权衡。

### 2. InnoDB 与 MyISAM 的区别 🔥

| 维度 | InnoDB | MyISAM |
| --- | --- | --- |
| 事务 | 支持（ACID） | 不支持 |
| 锁 | 行锁 + 表锁 | 仅表锁 |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 索引 | 聚簇索引（数据和索引在一起） | 非聚簇（索引和数据分离） |
| `count(*)` | 需扫描统计 | 维护了行数，O(1) |
| 适用 | **OLTP 默认（5.5+ 默认引擎）** | 只读/全文检索（已少用） |

## 二、索引（重中之重）

### 3. 索引是什么，优缺点 🔥

索引是提高查询效率的**数据结构**（以空间换时间）。优点：加速检索、加速排序分组、加速 join。缺点：占空间、降低写性能（增删改要维护索引）、需要维护。

### 4. 为什么用 B+Tree 而不用 B 树/红黑树/Hash 🔥🔥

- **红黑树/二叉树**：树高随数据量增长快，磁盘 IO 次数多（一次 IO 读一个节点）。
- **Hash 索引**：等值查询 O(1) 极快，但**不支持范围查询、排序、前缀匹配**。
- **B 树**：每个节点都存数据，单节点能存的 key 少，树更高。
- **B+Tree** 的优势：
  - **非叶子节点只存键**（不存数据），单页能存更多 key → **树更矮**（通常 3~4 层即可存千万级），IO 次数少（一般 ≤ 4 次）。
  - **叶子节点用双向链表相连** → 范围查询、排序高效。
  - 数据都在叶子节点，查询性能稳定。

### 5. 聚簇索引与非聚簇索引（二级索引）🔥

- **聚簇索引（主键索引）**：叶子节点存**整行数据**。InnoDB 一张表有且仅有一个聚簇索引（主键 → 唯一非空索引 → 隐藏 rowid）。
- **二级索引（辅助索引）**：叶子节点存**主键值**。通过二级索引查询非索引列需要拿主键再回聚簇索引查一次，即 **回表**。

### 6. 回表、覆盖索引、最左前缀 🔥🔥

- **回表**：二级索引查到主键后，再去聚簇索引取整行。
- **覆盖索引**：查询的列都在索引中，**无需回表**（`select` 的列被索引覆盖）。是重要优化手段。
- **最左前缀原则**：联合索引 `(a,b,c)` 能命中 `a`、`a,b`、`a,b,c`，不能命中 `b`、`b,c`（除非有 a 等值）。范围查询（`>`/`<`/`like`）后面的列无法用索引。

### 7. 索引失效的常见情况 🔥

1. 索引列上做**函数/运算/类型转换**（`where date(t)=...`、字符串字段传数字隐式转换）。
2. `like '%xx'` 以通配符开头。
3. 违反**最左前缀**。
4. `or` 连接的条件有非索引列。
5. 优化器判断**全表扫描更快**（区分度低/数据量小）。
6. `is null`/`!=`/`not in` 有时失效（视情况）。

> 用 `EXPLAIN` 看执行计划：关注 `type`（`const`>`ref`>`range`>`index`>`ALL`）、`key`（实际用的索引）、`rows`（扫描行数）、`Extra`（`Using index` 覆盖索引好、`Using filesort`/`Using temporary` 需优化）。

### 8. 索引设计/使用注意 ⭐

- 选择**区分度高**的列建索引。
- 联合索引把高频、高区分度、等值的列放左边。
- 控制索引数量（影响写）；用前缀索引控制长字符串索引大小。
- 主键用**自增整数**（顺序写、减少页分裂、二级索引体积小），避免用 UUID 等无序大主键。

## 三、事务与 MVCC

### 9. 事务的 ACID 与实现 🔥

- **原子性（A）**：`undo log`（回滚日志）保证失败可回滚。
- **一致性（C）**：是目的，由 AID + 业务约束共同保证。
- **隔离性（I）**：锁 + MVCC。
- **持久性（D）**：`redo log`（重做日志）保证提交后数据不丢（崩溃可恢复）。

### 10. 并发事务的问题与隔离级别 🔥

问题：**脏读**（读到未提交数据）、**不可重复读**（两次读同一行结果不同，因别的事务更新）、**幻读**（两次范围查询行数不同，因别的事务插入）。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
| --- | --- | --- | --- |
| 读未提交 | ✓ | ✓ | ✓ |
| 读已提交 RC | ✗ | ✓ | ✓ |
| 可重复读 RR（MySQL 默认） | ✗ | ✗ | ✗（基本解决） |
| 串行化 | ✗ | ✗ | ✗ |

**MySQL InnoDB 在 RR 下用 MVCC + 间隙锁（Next-Key Lock）基本解决了幻读**，这是它与标准 SQL 的差异。

### 11. MVCC（多版本并发控制）原理 🔥🔥

MVCC 让**读不加锁**（快照读），读写不阻塞，提升并发。核心三件套：

1. **隐藏字段**：每行有 `DB_TRX_ID`（最近修改的事务 ID）、`DB_ROLL_PTR`（回滚指针，指向 undo log 旧版本）。
2. **undo log 版本链**：每次修改生成新版本，通过回滚指针串成链。
3. **ReadView（一致性视图）**：记录生成视图时活跃（未提交）的事务 ID 列表。读取时沿版本链找到**对当前事务可见**的版本（判断该版本的 `DB_TRX_ID` 是否已提交/是否本事务所改）。

- **RC**：每次 `select` 都生成新的 ReadView → 能读到别的事务已提交的最新数据（不可重复读）。
- **RR**：事务内**第一次** `select` 生成 ReadView 并复用 → 同一事务多次读结果一致（可重复读）。

> **快照读**（普通 `select`，走 MVCC）vs **当前读**（`select ... for update`/`lock in share mode`、`update`/`delete`，读最新版本并加锁）。

### 12. MySQL 的锁 🔥

- 粒度：**表锁**（开销小、并发低）、**行锁**（InnoDB，开销大、并发高）、页锁。
- 行锁类型：
  - **记录锁（Record Lock）**：锁单行索引记录。
  - **间隙锁（Gap Lock）**：锁索引记录之间的间隙，防止插入（解决幻读）。
  - **临键锁（Next-Key Lock）**：记录锁 + 间隙锁，RR 默认。
- 共享锁 S（读锁）、排他锁 X（写锁）；意向锁 IS/IX（表级，加速判断）。
- **行锁是加在索引上的**：若查询走不到索引，会退化为**表锁**。

### 13. 乐观锁与悲观锁 ⭐

- **悲观锁**：假设会冲突，先加锁再操作（`select ... for update`）。
- **乐观锁**：假设不冲突，更新时用**版本号/CAS** 校验（`update t set v=v+1, version=version+1 where id=? and version=?`），冲突则重试。读多写少用乐观锁。

### 14. redo log、undo log、binlog 与两阶段提交 🔥

- **undo log**：回滚 + MVCC 版本链。
- **redo log**：InnoDB 引擎层，物理日志，崩溃恢复（WAL：先写日志再刷盘）。循环写。
- **binlog**：Server 层，逻辑日志，用于**主从复制**和数据恢复。追加写。
- **两阶段提交（2PC）**：更新时 redo log 先 `prepare`，写 binlog，再 redo log `commit`，保证 redo 和 binlog 一致（防止主从数据不一致）。

## 四、SQL 执行与优化

### 15. 一条查询 SQL 的执行流程 ⭐

`连接器（鉴权）→ 查询缓存（8.0 已移除）→ 分析器（词法/语法）→ 优化器（选索引、定执行计划）→ 执行器（调引擎接口）→ 存储引擎（返回数据）`。

### 16. SQL 优化手段 🔥

- 建合适索引、利用覆盖索引、避免回表。
- 避免 `select *`（减少 IO、利于覆盖索引）。
- 避免索引失效（见第 7 题）。
- 分页深翻优化：`limit 1000000,10` 慢 → 用**延迟关联**（先查主键再 join）或基于游标 `where id > ?`。
- 大事务拆小、批量操作。
- `join` 用小表驱动大表、被驱动表关联字段加索引。
- 用慢查询日志（`slow_query_log`）定位慢 SQL，`EXPLAIN` 分析。

### 17. `drop`/`delete`/`truncate` 区别 ⭐

| | `delete` | `truncate` | `drop` |
| --- | --- | --- | --- |
| 类型 | DML | DDL | DDL |
| 范围 | 可带 where | 清空整表 | 删表结构 |
| 事务 | 可回滚 | 不可回滚 | 不可回滚 |
| 速度 | 慢（逐行+日志） | 快 | 快 |
| 自增 | 不重置 | 重置 | — |

### 18. `count(*)` 在 InnoDB 比 MyISAM 慢的原因 ⭐

MyISAM 维护了表行数，`count(*)` 直接返回；InnoDB 因 MVCC 多版本，行数随事务而异，必须实时统计扫描。`count(*)`≈`count(1)`>`count(字段)`（字段要判 null），InnoDB 会选最小的二级索引扫描。

### 19. `varchar` vs `char`，`varchar(30)`、`int(11)` 含义 ⭐

- `char(n)` 定长（不足补空格），`varchar(n)` 变长（省空间，需额外长度字节）。
- `varchar(30)` 的 30 是**字符数上限**，不是字节。
- `int(11)` 的 11 是**显示宽度**（配合 `zerofill` 补零），**不影响存储范围**，`int` 始终 4 字节。8.0 起该特性废弃。

## 五、分库分表与高可用

### 20. 大表如何优化 🔥

由轻到重：
1. SQL 与索引优化、读写分离（主写从读）。
2. 缓存（Redis）挡读流量。
3. **垂直拆分**（按字段/业务拆表拆库）。
4. **水平拆分（分库分表）**：按 range / hash / 一致性哈希分片，工具如 ShardingSphere、MyCat。
5. 冷热分离、归档历史数据。

### 21. 分库分表后主键怎么处理 🔥

自增主键会冲突，需**全局唯一 ID**：
- **UUID**：无序、占空间、不利于索引，不推荐做主键。
- **数据库号段**（多台步长不同）。
- **Snowflake 雪花算法**：64 位 = 时间戳 + 机器 ID + 序列号，趋势递增、高性能，注意**时钟回拨**问题。
- **Redis incr / 美团 Leaf / 百度 UidGenerator**。

### 22. 分库分表带来的问题 ⭐

跨库 join 困难、分布式事务、跨库分页/排序/聚合复杂、全局 ID、扩容数据迁移。所以**能不分就不分**，先穷尽单库优化（读写分离、缓存、归档）。

### 23. MySQL 高可用方案 ⭐

- **主从复制**（binlog → relay log → 重放），读写分离。
- **MHA / MGR（MySQL Group Replication）** 自动故障切换。
- 一主多从、双主、级联复制；配合中间件（ShardingSphere-Proxy）做读写分离与分片。
- 注意**主从延迟**：可强制读主、半同步复制、并行复制缓解。
