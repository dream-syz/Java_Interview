# 04 · 并发编程

> 覆盖：线程基础、JMM 与 `volatile`、`synchronized` 与锁升级、`ReentrantLock` 与 AQS、CAS、`ThreadLocal`、线程池、并发容器与工具、死锁，以及 JDK 21 虚拟线程。**面试占比最高、区分度最大的模块。**

## 一、线程基础

### 1. 创建线程的方式 🔥

1. 继承 `Thread`，重写 `run()`；
2. 实现 `Runnable`（推荐，避免单继承限制）；
3. 实现 `Callable` + `FutureTask`（有返回值、可抛异常）；
4. 线程池 `ExecutorService`（**生产中唯一推荐**，复用线程、可控）。

> 本质上只有一种：实现 `Runnable`/`Callable` 任务交给线程执行。`Thread.start()` 才会真正创建线程并调度 `run()`；直接调 `run()` 只是普通方法调用，在当前线程执行。

### 2. 线程的状态 🔥

`NEW → RUNNABLE →（BLOCKED / WAITING / TIMED_WAITING）→ TERMINATED`。

- `BLOCKED`：等待 `synchronized` 锁。
- `WAITING`：`wait()`/`join()`/`park()` 无限等待。
- `TIMED_WAITING`：带超时的等待（`sleep`、`wait(t)`）。

### 3. `sleep` 与 `wait` 的区别 🔥

| | `sleep` | `wait` |
| --- | --- | --- |
| 所属 | `Thread`（静态） | `Object` |
| 释放锁 | **不释放** | **释放**对象锁 |
| 唤醒 | 时间到自动醒 | `notify`/`notifyAll` 或超时 |
| 使用位置 | 任意 | 必须在 `synchronized` 块内 |

### 4. 为什么 `wait`/`notify` 定义在 `Object` 而非 `Thread` ⭐

因为锁是绑定在**任意对象**上的（每个对象都有 monitor），`wait`/`notify` 操作的是“对象的监视器队列”，所以必须定义在 `Object`。同理它们必须在 `synchronized` 块内调用（否则抛 `IllegalMonitorStateException`），因为要先持有该对象锁。

### 5. `notify` 与 `notifyAll` 的区别 ⭐

`notify` 随机唤醒一个等待线程，`notifyAll` 唤醒所有（再竞争锁）。生产建议用 `notifyAll` 避免信号丢失，且 `wait` 要放在 **while 循环**里防止虚假唤醒。

### 6. 如何优雅停止线程 🔥

- **不要用** `stop()`（已废弃，强制终止会破坏一致性、不释放锁）。
- 正确做法：**中断协作**。`interrupt()` 设置中断标志，任务内周期性检查 `Thread.currentThread().isInterrupted()` 或捕获 `InterruptedException` 后退出。
- `interrupted()`（静态，检查并**清除**标志）vs `isInterrupted()`（实例，仅检查不清除）。

## 二、JMM 与 volatile

### 7. JMM（Java 内存模型）是什么，为什么需要 🔥

JMM 定义了多线程下**共享变量的可见性、有序性**规则，屏蔽底层 CPU/缓存/编译器差异。核心概念：

- 每个线程有**工作内存**（对应寄存器/缓存），共享变量在**主内存**；线程操作的是副本，需同步回主内存。
- **三大特性**：原子性、可见性、有序性。
- **happens-before** 规则：定义了操作间的偏序关系（程序顺序、锁、`volatile`、线程启动/终止、传递性等），是判断是否存在数据竞争的依据。

### 8. `volatile` 的作用与原理 🔥

保证两点（**不保证原子性**）：

1. **可见性**：写 `volatile` 立即刷主内存，读 `volatile` 从主内存读。
2. **有序性**：禁止指令重排序（插入**内存屏障**）。

- 底层：写后插 StoreStore/StoreLoad 屏障，读前插 LoadLoad/LoadStore 屏障；汇编层是 `lock` 前缀。
- 经典用途：**双重检查锁单例**的 `volatile` 修饰，防止 `new` 的指令重排导致拿到未初始化对象。
- `i++` 这种复合操作 `volatile` 保证不了原子性，需 `synchronized` 或 `AtomicInteger`。

```java
public class Singleton {
    private static volatile Singleton instance; // volatile 必需
    public static Singleton get() {
        if (instance == null) {                 // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {         // 第二次检查
                    instance = new Singleton(); // 防重排：分配→构造→赋引用
                }
            }
        }
        return instance;
    }
}
```

## 三、synchronized 与锁

### 9. `synchronized` 的原理与锁升级 🔥

- 字节码：同步块用 `monitorenter`/`monitorexit`，同步方法用 `ACC_SYNCHRONIZED` 标志。底层是对象监视器 **Monitor（ObjectMonitor）**。
- **锁升级**（JDK 6 引入，状态存在对象头 Mark Word）：
  - **无锁 → 偏向锁**：只有一个线程访问时，记录线程 ID，无需 CAS。
  - **→ 轻量级锁**：有竞争时，用 CAS 自旋尝试获取（适合锁持有时间短）。
  - **→ 重量级锁**：自旋失败、竞争激烈，升级为重量级锁，线程进入阻塞（涉及内核态切换）。
  - 锁只能升级不能降级。
- 注：**偏向锁在 JDK 15 默认关闭、JDK 18 移除**（维护成本高、收益低）。

### 10. `synchronized` 与 `ReentrantLock` 的区别 🔥

| 维度 | `synchronized` | `ReentrantLock` |
| --- | --- | --- |
| 层面 | JVM 关键字 | JDK 类（AQS 实现） |
| 加解锁 | 自动 | 手动 `lock()`/`unlock()`（须 finally 释放） |
| 公平性 | 非公平 | 可选公平/非公平 |
| 可中断 | 否 | `lockInterruptibly()` |
| 超时 | 否 | `tryLock(timeout)` |
| 条件 | 单一 wait set | 多个 `Condition` |
| 性能 | 优化后接近 | 灵活 |

> 选型：默认 `synchronized`（简单、自动释放、JVM 持续优化）；需要可中断/超时/公平/多条件队列时用 `ReentrantLock`。

### 11. 锁优化机制 ⭐

自旋锁与自适应自旋、锁消除（逃逸分析）、锁粗化（连续加锁合并）、偏向/轻量级锁升级、读写分离（`ReentrantReadWriteLock`/`StampedLock`）。

## 四、AQS 与 CAS

### 12. AQS（AbstractQueuedSynchronizer）是什么 🔥

AQS 是 `ReentrantLock`、`Semaphore`、`CountDownLatch`、`ReentrantReadWriteLock` 等的**底层框架**。核心：

- 一个 **`volatile int state`** 表示同步状态（如重入次数、许可数）。
- 一个 **CLH 变体的双向队列（FIFO）** 存放被阻塞线程，用 `LockSupport.park/unpark` 阻塞唤醒。
- 通过 **CAS** 修改 state 实现无锁竞争入口。
- 模板方法：子类实现 `tryAcquire`/`tryRelease`（独占）或 `tryAcquireShared`（共享）。

### 13. CAS 原理与缺点 🔥

- **CAS（Compare-And-Swap）**：`(内存值, 期望值, 新值)`，当内存值等于期望值时才更新，是一条 CPU 原子指令（`cmpxchg`），是乐观锁/无锁结构的基础。
- 缺点：
  1. **ABA 问题**：值从 A→B→A，CAS 误以为没变。解决：加版本号 `AtomicStampedReference`。
  2. **自旋开销**：竞争激烈时长时间自旋耗 CPU。
  3. **只能保证单变量原子性**：多变量需 `AtomicReference` 封装或加锁。
- `AtomicInteger` 等基于 `Unsafe` 的 CAS；高并发计数推荐 **`LongAdder`**（分段累加，减少竞争）。

## 五、ThreadLocal

### 14. `ThreadLocal` 原理 🔥

- 每个 `Thread` 内部有一个 `ThreadLocalMap`，key 是 `ThreadLocal`（**弱引用**），value 是值。变量对每个线程独立，实现线程隔离。
- **内存泄漏风险**：key 是弱引用会被回收变成 null，但 value 是强引用仍在 Map 中。**用完务必 `remove()`**，尤其在**线程池**中（线程复用，残留脏数据）。
- 父子线程传递用 `InheritableThreadLocal`；线程池场景用阿里 **TransmittableThreadLocal（TTL）**。
- 典型用途：数据库连接/Session、用户上下文（如 `RequestContextHolder`）、`SimpleDateFormat` 线程安全化。

## 六、线程池

### 15. 线程池的核心参数与执行流程 🔥

`ThreadPoolExecutor` 七参数：

```java
new ThreadPoolExecutor(
    corePoolSize,      // 核心线程数
    maximumPoolSize,   // 最大线程数
    keepAliveTime,     // 非核心线程空闲存活时间
    unit,
    workQueue,         // 任务队列
    threadFactory,     // 线程工厂（建议自定义命名，便于排查）
    handler);          // 拒绝策略
```

**执行流程**：

1. 线程数 < core → 新建核心线程执行；
2. 否则入**队列**；
3. 队列满且线程数 < max → 新建非核心线程；
4. 都满 → 执行**拒绝策略**。

### 16. 拒绝策略 ⭐

- `AbortPolicy`（默认，抛 `RejectedExecutionException`）；
- `CallerRunsPolicy`（调用线程自己执行，起到反压/降速作用）；
- `DiscardPolicy`（静默丢弃）；
- `DiscardOldestPolicy`（丢弃最老任务）。
- 生产常**自定义**：落库/告警/降级。

### 17. 为什么不建议用 `Executors` 创建线程池 🔥

阿里规约：`Executors` 隐藏风险——

- `newFixedThreadPool`/`newSingleThreadExecutor` 用**无界队列** `LinkedBlockingQueue`，任务堆积 → OOM。
- `newCachedThreadPool`/`newScheduledThreadPool` 允许创建 `Integer.MAX_VALUE` 个线程 → OOM。
- **应手动 `new ThreadPoolExecutor`**，明确各参数与有界队列。

### 18. 核心线程数怎么设置 🔥

- **CPU 密集型**：核心数 ≈ `CPU核数 + 1`（减少上下文切换）。
- **IO 密集型**：核心数 ≈ `CPU核数 × (1 + 平均等待时间/平均计算时间)`，经验值 `2 × CPU核数` 起步。
- 实际应结合压测与监控动态调整，可做成**可动态调参的线程池**（如美团 DynamicTp 思路）。

### 19. `submit` 与 `execute` 的区别 ⭐

- `execute(Runnable)`：无返回值，异常直接抛到线程（可被 `UncaughtExceptionHandler` 捕获）。
- `submit(...)`：返回 `Future`，**异常被封装在 Future 中**，调用 `get()` 才抛出（容易吞异常，注意处理）。

### 20. 常见阻塞队列 ⭐

`ArrayBlockingQueue`（有界数组）、`LinkedBlockingQueue`（可选有界链表）、`SynchronousQueue`（不存元素，直接移交，`CachedThreadPool` 用）、`PriorityBlockingQueue`（优先级）、`DelayQueue`（延迟）。

## 七、并发工具与容器

### 21. `CountDownLatch` vs `CyclicBarrier` vs `Semaphore` ⭐

- **CountDownLatch**：一次性，`await` 等待计数归零（如主线程等多个子任务完成）。
- **CyclicBarrier**：可重用，多个线程互相等待到达屏障再一起继续（如多阶段并行计算）。
- **Semaphore**：信号量，控制同时访问的线程数（如限流、连接池）。

### 22. `Callable` / `Future` / `CompletableFuture` ⭐

- `Future` 只能阻塞 `get()` 或轮询，能力弱。
- **`CompletableFuture`（JDK 8）** 支持链式编排、组合（`thenApply`/`thenCompose`/`allOf`）、异常处理（`exceptionally`），是异步编排首选。

### 23. `CopyOnWriteArrayList` 的应用场景 ⭐

写时复制：写操作复制整个数组（加锁），读操作无锁。适合**读多写极少**（如监听器列表、白名单），不适合写频繁（每次写都全量拷贝）。

### 24. 阻塞队列实现生产者-消费者 ⭐

`BlockingQueue` 的 `put`（队满阻塞）/`take`（队空阻塞）天然实现生产者-消费者，无需手写 `wait/notify`。

## 八、死锁与三线程顺序执行

### 25. 死锁的四个必要条件与避免 🔥

四条件（同时满足才死锁）：**互斥、请求并保持、不可剥夺、循环等待**。

避免：

- **破坏循环等待**：按固定全局顺序申请锁（最常用）。
- 破坏请求并保持：一次性申请所有资源。
- 用 `tryLock(timeout)` 超时放弃。
- 检测：`jstack` 会直接报告 `Found one Java-level deadlock`。

### 26. 三个线程 T1、T2、T3 如何保证顺序执行 ⭐

最简单：用 `t.join()`，让后一个线程在前一个线程结束后再启动。也可用 `CountDownLatch`、单线程池、`CompletableFuture` 串行编排。

```java
t1.start(); t1.join();
t2.start(); t2.join();
t3.start();
```

## 九、虚拟线程（JDK 21）🆕

### 27. 虚拟线程 Virtual Threads

- 轻量级线程，由 JVM 调度到少量平台线程（载体线程）上，创建成本极低（可百万级）。
- 解决传统“一请求一线程”在高并发 IO 下线程爆炸的问题，**用同步阻塞的写法获得异步的吞吐**。
- 适合 **IO 密集型**（大量阻塞等待），CPU 密集型收益不大。
- 注意：`synchronized` 内阻塞会“钉住（pinning）”载体线程，推荐用 `ReentrantLock`；不要池化虚拟线程。

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10_000).forEach(i ->
        executor.submit(() -> { Thread.sleep(1000); return i; }));
}
```
