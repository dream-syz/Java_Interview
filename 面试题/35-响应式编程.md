# 35 · 响应式编程（Reactive Programming）

> 响应式是高级岗的**加分项**，也是容易被反问「虚拟线程出来后响应式还有用吗」的话题。本篇讲清：响应式是什么、为什么、Reactor 核心（Flux/Mono/背压）、WebFlux、线程模型、与传统/虚拟线程的对比，以及**什么时候不该用**。

---

## 一、响应式是什么 & 为什么 🔥

### 1. 定义

响应式编程是一种**基于异步数据流 + 非阻塞 + 背压** 的编程范式。核心是 **Reactive Streams 规范**（`Publisher`/`Subscriber`/`Subscription`/`Processor` 四接口）。

### 2. 解决什么问题（动机）⭐

传统 **thread-per-request**（一请求一线程，Servlet 阻塞模型）的痛点：

- 每个请求独占线程，**线程阻塞在 IO 等待**（调下游、查库）时白白占着线程。
- 高并发下线程数爆炸 → 内存（每线程 1MB 栈）+ 上下文切换开销大（见 [32-操作系统](./32-操作系统.md)）。

响应式用**少量线程（通常 = CPU 核数）处理大量并发连接**，IO 等待时线程去干别的活（事件驱动），**资源利用率高、抗高并发**。

### 3. 响应式宣言四特质

**响应性（Responsive）、弹性（Resilient，容错）、回弹性（Elastic，可伸缩）、消息驱动（Message Driven）**。

## 二、Reactor 核心 🔥（Spring 生态的响应式库）

### 4. 两个核心类型

| 类型 | 含义 |
| --- | --- |
| **`Mono<T>`** | 0 或 1 个元素的异步序列（如单次查询） |
| **`Flux<T>`** | 0 到 N 个元素的异步流（如列表/流式数据） |

```java
Flux.just(1, 2, 3)
    .map(i -> i * 2)              // 转换
    .filter(i -> i > 2)          // 过滤
    .flatMap(i -> callRemote(i)) // 异步展开（返回 Publisher）
    .subscribe(System.out::println);  // 订阅才真正执行（Lazy）
```

### 5. 关键特性 ⭐

- **Lazy（惰性）**：不 `subscribe()` 就不执行（声明的是「流水线蓝图」，订阅才触发）。
- **操作符**：map/filter/flatMap/zip/merge/reduce/buffer/window…（类似 Stream 但异步）。
- **错误处理**：`onErrorReturn`/`onErrorResume`/`retry`（流式容错）。
- **冷流 vs 热流**：冷流每个订阅者独立从头消费；热流共享，订阅后才收到后续数据。

### 6. 背压（Backpressure）🔥🔥（核心考点）

**消费者处理不过来时，能反向告诉生产者「慢点发」**，防止快生产者压垮慢消费者导致 OOM。

- `Subscriber` 通过 `request(n)` 告诉上游「我只要 n 个」，上游按需推送（**pull-push 结合**）。
- 处理策略（生产快于消费时）：`BUFFER`（缓冲，可能 OOM）、`DROP`（丢新）、`LATEST`（只留最新）、`ERROR`（报错）。

> 这是响应式相比普通异步回调（Future/CompletableFuture）的**关键差异**：内置了流量控制。

## 三、WebFlux 与线程模型 🔥

### 7. Spring MVC vs WebFlux

| 维度 | Spring MVC | Spring WebFlux |
| --- | --- | --- |
| 模型 | 同步阻塞，thread-per-request | 异步非阻塞，事件驱动 |
| 容器 | Tomcat（Servlet） | **Netty**（默认）/ Undertow |
| 线程 | 大线程池 | 少量 EventLoop 线程 |
| 返回值 | `User` / `List` | `Mono<User>` / `Flux<User>` |
| 适用 | CPU 密集 / 普通业务 | **IO 密集、高并发、流式、网关** |

### 8. 线程模型（Netty EventLoop）

- 基于 **Reactor 模式**（见 [28-Netty](./28-Netty与网络编程.md)）：少量 EventLoop 线程跑事件循环。
- **绝对铁律：EventLoop 线程上不能有任何阻塞操作**（阻塞 JDBC、`Thread.sleep`、同步 IO），否则整个事件循环卡死，吞吐崩塌。
- 阻塞操作必须切到弹性线程池：`subscribeOn(Schedulers.boundedElastic())`。
- 全链路非阻塞才有意义：要用 **R2DBC**（响应式数据库驱动）、响应式 Redis/WebClient，否则中间一个阻塞调用就破功。

### 9. Schedulers（调度器）⭐

| Scheduler | 用途 |
| --- | --- |
| `parallel()` | CPU 密集（固定 = 核数） |
| `boundedElastic()` | 包裹阻塞 IO（有界弹性线程池） |
| `single()` | 单线程 |
| `immediate()` | 当前线程 |

`subscribeOn` 影响整条链的执行线程（源头），`publishOn` 切换其后操作符的线程。

## 四、与虚拟线程的对比 🔥🔥（最热反问）

> 「JDK 21 虚拟线程（见 [04-并发编程](./04-并发编程.md)）出来后，响应式还有必要吗？」

| 维度 | 响应式（WebFlux） | 虚拟线程（Loom） |
| --- | --- | --- |
| 编程模型 | 函数式、声明式，**学习曲线陡，调试难**（栈不直观） | **同步阻塞写法**，像普通代码，易读易调 |
| 实现 | 应用层事件循环 | JVM 层调度，遇阻塞自动让出载体线程 |
| 资源 | 少线程高并发 | 海量轻量虚拟线程 |
| 背压 | **内置背压** | 无原生背压（需自己控流） |
| 适用 | 流式处理、需要背压、复杂数据流编排 | 普通高并发 IO 业务（绝大多数场景） |

**结论（面试金句）**：

- 虚拟线程让「用简单的同步写法获得高并发」成为可能，**抢走了响应式大部分「省线程」的理由**，多数 IO 密集业务用虚拟线程更划算。
- 响应式仍有不可替代的价值：**背压、流式数据处理、丰富的声明式操作符编排**（如实时数据流、SSE、网关）。
- 趋势：普通业务 → 虚拟线程；数据流/流处理/需要背压 → 响应式。

---

## 五、什么时候不该用响应式 ⭐

- **简单 CRUD / 低并发**：收益小，徒增复杂度和调试成本。
- **团队不熟悉**：响应式心智负担重，排错难（堆栈断裂、线程切换）。
- **链路中有无法非阻塞的依赖**（老的阻塞 JDBC/SDK）：一个阻塞点就毁掉整条链。
- 有了**虚拟线程**后，很多原本「为了抗并发上响应式」的场景可以回归简单写法。

## 高频追问清单

- 响应式解决什么问题？→ 少线程抗高并发 IO，省线程切换（一）。
- 背压是什么，怎么实现？→ 消费者 request(n) 反压上游（二）。
- WebFlux 为什么不能在 EventLoop 上阻塞？→ 卡死事件循环（三）。
- Mono 和 Flux 区别？→ 0-1 vs 0-N（二）。
- 虚拟线程出来后响应式还有用吗？→ 背压/流式仍不可替代，普通业务用虚拟线程（四）。
- 为什么响应式必须全链路非阻塞？→ 一个阻塞点破功，需 R2DBC（三）。
