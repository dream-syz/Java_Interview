# 21 · ElasticSearch

> 覆盖：是什么、倒排索引、核心概念、写入与查询流程、分片与副本、近实时、深分页、与数据库一致性、调优。

---

## 1. ElasticSearch 是什么 🔥

基于 **Lucene** 的分布式搜索与分析引擎，擅长**全文检索**、海量数据近实时检索与聚合分析。常用于日志分析（ELK）、搜索、数据分析。

## 2. 核心概念 ⭐

| ES 概念 | 类比（旧） | 说明 |
| --- | --- | --- |
| Index | 数据库 | 索引（7.x 后一个 index 只有一个 type `_doc`） |
| Document | 行 | JSON 文档 |
| Field | 列 | 字段 |
| Mapping | 表结构 | 字段类型定义 |
| Shard | 分片 | 索引水平切分，分布到节点 |
| Replica | 副本 | 分片的复制，高可用 + 提升查询 |

## 3. 倒排索引（核心）🔥🔥

传统正排索引是“文档 → 词”，倒排索引是“**词 → 文档列表**”：

- **分词**：把文本切成词条（Term），经过分析器（analyzer：字符过滤 → 分词 → 词条过滤如小写/停用词/词干）。
- **词典（Term Dictionary）+ 倒排列表（Posting List）**：每个词指向包含它的文档 ID 列表 + 词频/位置。
- 查询时：对查询词分词 → 在词典找到词 → 取倒排列表 → 求交并集 → 打分排序（TF-IDF / BM25）。
- 词典用 **FST（有限状态转换器）** 压缩存储，省内存、查询快。

## 4. 写入流程 🔥

1. 文档先写**内存 buffer** + **translog（事务日志，防丢）**。
2. 默认每 **1 秒** `refresh`：buffer 数据生成新的 **segment（段）** 并进入 OS Cache，此时**可被搜索**（这就是**近实时 NRT** 的来源）。
3. 默认每 30 分钟或 translog 满 `flush`：segment 落盘，清空 translog。
4. 后台 `merge` 把小段合并成大段（删除标记的文档此时真正物理删除）。

## 5. 查询流程（Query then Fetch）⭐

1. **Query 阶段**：协调节点把请求广播到相关分片，各分片本地查询返回**文档 ID + 打分**，协调节点汇总排序得到 TopN 的 ID。
2. **Fetch 阶段**：协调节点向相关分片拉取这些 ID 的完整文档，返回客户端。

## 6. 分片与副本 🔥

- **主分片（primary）**：数据写入的分片，数量**创建后不可改**（涉及路由 `hash(_id) % primary_count`）。
- **副本分片（replica）**：主分片的拷贝，可动态调整，提供高可用和读扩展。
- 写走主分片再同步副本；读可走主或副本。

## 7. 深分页问题 🔥

`from + size` 深翻（如 `from=10000`）需每个分片取 `from+size` 再汇总，**代价随页数剧增**，ES 默认限制 `from+size ≤ 10000`。解决：

- **search_after**：基于上一页最后一条的排序值翻页（推荐，无状态、适合实时翻页）。
- **scroll**：快照游标，适合**导出大量数据**（非实时，占资源）。
- **PIT（Point In Time）+ search_after**：7.10+ 推荐方案。

## 8. ES 与数据库的数据一致性 ⭐

ES 通常作为**辅助检索**，主数据在 MySQL。同步方式：

- 双写（代码同时写 DB 和 ES，简单但不可靠）。
- **MQ 异步同步**。
- **Canal 订阅 binlog** 同步（解耦、可靠，推荐）。
- 接受**最终一致**，定时全量对账兜底。

## 9. 调优要点 ⭐

- 合理设置分片数（避免过多小分片）、副本数。
- 写入：批量 `bulk`、增大 `refresh_interval`、关闭不需要的 `_source`/字段。
- 查询：用 filter（可缓存、不算分）替代 query、避免深分页、控制聚合基数。
- 冷热分离、ILM 索引生命周期管理。
- 合理 mapping：不需要分词的字段用 `keyword`，控制 `text` 字段。
