# 31 · 项目与软技能

> 资深 / 架构师面试有近一半时间在聊项目、技术选型、架构演进和软技能。这部分没有标准答案，但有**套路和加分项**。把它和前面的硬核知识结合，才是完整的资深画像。

---

## 一、如何讲项目（STAR 法则）🔥

用 **STAR** 结构讲，逻辑清晰、有说服力：

- **S（Situation 背景）**：项目是做什么的、规模（用户量/QPS/数据量）、你的角色。
- **T（Task 任务）**：你要解决的具体问题/挑战。
- **A（Action 行动）**：你**怎么做的**（技术方案、为什么这么选、踩了什么坑）—— 这是重点，体现你的深度。
- **R（Result 结果）**：**量化结果**（响应时间从 2s 降到 200ms、QPS 提升 5 倍、成本下降 30%）。

> 大忌：只说"我用了 XX 技术"，不说为什么、不说效果。面试官要听的是**你的思考和取舍**，不是技术名词堆砌。

### 讲项目的准备清单

- 项目整体架构图（能手画）。
- 你负责的核心模块及难点。
- 至少 2~3 个能深挖的技术亮点（你主导设计、有量化收益的）。
- 踩过的坑和怎么解决的（线上事故复盘最能体现能力）。

---

## 二、技术选型怎么答 🔥

面试官问"你为什么用 Redis 不用 XX""为什么 RocketMQ 不用 Kafka"，考察的是**权衡能力**，不是对错。

回答框架：

1. **业务需求驱动**：先说业务场景的约束（一致性要求、吞吐、延迟、团队熟悉度、成本）。
2. **候选方案对比**：列 2~3 个候选，对比优劣。
3. **结论 + 取舍**：基于需求选哪个，**牺牲了什么**（没有银弹）。

> 例："我们选 RocketMQ 而非 Kafka，因为业务需要**事务消息和顺序消息**（订单场景），Kafka 这方面支持弱；代价是 RocketMQ 极限吞吐不如 Kafka，但我们的量级（万级 TPS）完全够用。"

## 三、架构演进故事 ⭐

资深/架构师常被问"系统怎么从小做大的"。经典演进路径（能讲清楚就是加分）：

```
单体应用
 → 垂直拆分（按模块分应用 + 分库）
 → 引入缓存 / 读写分离 / CDN（扛读）
 → 引入 MQ（解耦 + 削峰）
 → 微服务化（服务拆分 + 注册中心 + 网关 + 配置中心）
 → 分库分表（扛写/存储）
 → 服务治理（熔断/限流/链路追踪）
 → 容器化 + K8s + 弹性伸缩
 → 单元化 / 异地多活（容灾 + 突破容量上限）
```

讲的时候要点出**每一步是被什么瓶颈逼出来的**（量变引起质变），而不是为了用新技术而用。**不要过度设计**——架构是演进出来的，不是一开始就堆上所有组件。

## 四、线上事故复盘（高含金量）⭐

准备 1~2 个真实事故，按这个结构讲：

1. **现象**：什么时间、什么影响（多少用户受影响、持续多久）。
2. **定位**：怎么排查的（监控告警 → 链路 → 日志 → JVM，见 [25-可观测性](./25-可观测性.md)）。
3. **根因**：真正原因（如缓存击穿、慢 SQL、线程池满、Full GC、死锁）。
4. **解决**：临时止血 + 根治。
5. **改进**：怎么避免复发（加监控、限流、预案、Code Review、混沌演练）。

> 能讲清楚一次完整的事故复盘，比背 100 道八股更能证明你是资深。

## 五、架构设计原则 ⭐

被问"你的架构设计理念"时可以提：

- **高内聚低耦合**、**面向接口/抽象**。
- **简单优先（KISS）、不过度设计（YAGNI）**。
- **可演进性 > 一步到位**：架构要能随业务增长平滑演进。
- **可观测、可降级、可回滚**：默认系统会出问题，要能发现、能兜底、能恢复。
- **成本意识**：技术方案要考虑机器成本、人力维护成本，不是越炫越好。
- **数据驱动**：靠压测和监控数据做决策，而非拍脑袋。

---

## 六、软技能与反问 ⭐

- **沟通**：能把复杂技术讲给非技术的人听（讲清"为什么"）。
- **推动力**：跨团队协作、技术方案落地的推动。
- **学习能力**：怎么跟进新技术、源码阅读习惯。
- **带人/影响力**（架构师/Leader 岗）：技术评审、规范制定、梯队培养。

### 反问环节（别浪费）

面试最后的反问也是加分项，可问：团队技术栈和挑战、这个岗位最想解决的问题、技术决策流程、成长空间。展现你的主动性和匹配度。

---

## 高频追问清单

- 讲一个你最有成就感的项目？→ STAR + 量化结果 + 技术亮点（一）。
- 为什么选 X 不选 Y？→ 需求驱动 + 对比 + 取舍（二）。
- 系统怎么支撑流量增长的？→ 架构演进，每步对应瓶颈（三）。
- 讲一次线上故障？→ 现象→定位→根因→解决→改进（四）。
