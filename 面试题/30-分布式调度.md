# 30 · 分布式调度（定时任务）

> 覆盖：单机定时任务方案、为什么需要分布式调度、XXL-JOB / ElasticJob 原理、任务分片、调度高可用、与延迟任务的区别。

---

## 1. 单机定时任务方案 ⭐

| 方案 | 说明 | 局限 |
| --- | --- | --- |
| `Thread.sleep` 循环 | 最原始 | 不可靠 |
| `Timer` | JDK 自带 | 单线程，一个任务异常影响其它 |
| `ScheduledThreadPoolExecutor` | 线程池版 | 推荐的单机方案 |
| Spring `@Scheduled` | 注解式，cron 表达式 | 默认单线程、单机 |

---

## 2. 为什么需要分布式调度 🔥

集群部署时，`@Scheduled` 会在**每个节点都执行一次**，导致任务**重复执行**。需要解决：

- **避免重复执行**（同一任务集群只跑一次）。
- **高可用**（某节点挂了任务仍能执行）。
- **任务分片**（大任务拆分到多节点并行）。
- **可视化管理 + 失败重试 + 监控告警**。

简单方案：分布式锁（Redis/ZK）保证只有一个节点执行；专业方案：XXL-JOB、ElasticJob、PowerJob。

---

## 3. XXL-JOB 原理 🔥（最常用）

**中心化调度**架构，分两部分：

- **调度中心（admin）**：负责任务管理、触发调度、监控（自身可集群部署，DB 做高可用，用 DB 行锁/时间轮避免重复调度）。
- **执行器（executor）**：嵌入业务应用，注册到调度中心，接收调度请求执行任务。

流程：调度中心按 cron 到点 → 根据路由策略选执行器 → RPC 触发执行 → 执行器回调结果。

特点：

- **调度与执行分离**：调度中心不执行业务，只负责"什么时候、让谁执行"。
- **路由策略**：第一个、轮询、随机、一致性 hash、最不经常使用、故障转移、忙碌转移、分片广播等。
- **故障转移**：执行失败转移到其它执行器。
- **分片广播**：把任务广播到所有执行器，每个分片处理一部分数据（分片参数 `shardIndex`/`shardTotal`）。

---

## 4. ElasticJob 原理 ⭐

**去中心化**，基于 **Zookeeper** 协调：

- 用 ZK 做注册中心 + 分布式协调（选主、分片分配、节点上下线感知）。
- **作业分片**：把任务分成多个分片项，ZK 负责把分片分配给各节点，节点只处理分到的分片，实现并行处理。
- 主节点（ZK 选举）负责分片，节点变化时重新分片（re-sharding）。
- 弹性：增减节点时自动重新分配分片。

对比：XXL-JOB 中心化、上手简单、有管理后台；ElasticJob 去中心化、依赖 ZK、分片能力强。

---

## 5. 任务分片的意义 🔥

单机处理百万数据慢，分片把数据按规则（如 `id % 分片数`）拆给多个节点并行处理，大幅提速。每个节点根据分配到的 `shardIndex` 处理对应数据段。

---

## 6. 分布式调度 vs 延迟任务 ⭐

- **分布式调度**：**周期性/定时**执行（每天 0 点对账、每 5 分钟同步）。
- **延迟任务**：**一次性**、到点执行（订单 30 分钟关闭）→ 用 MQ 延迟消息 / 时间轮（见 [23-系统设计](./23-系统设计.md)）。

---

## 7. 时间轮算法（TimingWheel）⭐（原理深挖）

大量定时任务若用 `ScheduledThreadPoolExecutor`（最小堆，插入/取出 O(logN)），海量任务下性能差。**时间轮**把时间分成 N 个槽（slot）组成环形数组，指针每隔固定 tick 前进一格，**插入/删除 O(1)**：

- 任务按到期时间哈希到对应槽的链表；指针转到该槽就执行该槽所有任务。
- 超过一圈的任务记 `round`（转几圈后执行），或用**分层时间轮**（秒轮→分轮→时轮，类似钟表进位）解决长延迟。
- 应用：**Netty `HashedWheelTimer`**、Kafka 延迟操作、时间轮做订单超时（见 [23-系统设计](./23-系统设计.md)）。
- 代价：是近似定时（精度取决于 tick），不适合要求精确到毫秒的强实时场景。

## 8. 调度的幂等与漏跑/重跑 🔥（实战）

- **重复触发**：网络抖动/重试可能让任务跑两次 → 任务逻辑必须**幂等**（处理前判状态、用唯一执行 ID 去重，见 [15-分布式](./15-分布式.md)）。
- **漏跑补偿**：节点宕机、调度中心故障期间错过的任务，需要**补偿机制**（记录任务执行台账，启动时扫描未执行的任务补跑）。
- **任务堆积**：上一次还没跑完下一次又到点 → 配置**阻塞处理策略**（XXL-JOB：单机串行/丢弃后续/覆盖之前），避免任务雪崩。
- **大任务超时**：拆分片 + 设超时 + 分片失败重试，避免一个慢分片拖垮整体。

## 9. 选型对比 ⭐

| 框架 | 架构 | 依赖 | 特点 |
| --- | --- | --- | --- |
| `@Scheduled`+分布式锁 | — | Redis/ZK | 最简，无管理界面、无分片 |
| **XXL-JOB** | 中心化 | MySQL | 上手快、有后台、路由/分片/故障转移，最流行 |
| **ElasticJob** | 去中心化 | ZooKeeper | 分片能力强、弹性扩容 |
| **PowerJob** | 中心化 | MySQL | 更现代，支持 MapReduce 式任务、工作流编排 |

## 高频追问清单

- 集群下 `@Scheduled` 会重复执行怎么办？→ 分布式锁 / 分布式调度框架（2）。
- XXL-JOB 怎么保证调度中心不重复触发？→ DB 锁 + 集群调度协调（3）。
- 大数据量定时任务怎么加速？→ 任务分片并行（5）。
- 调度框架怎么做高可用？→ 调度中心集群 + 执行器故障转移（3/4）。
- 海量定时任务用什么数据结构？→ 时间轮 O(1)，Netty HashedWheelTimer（7）。
- 任务被触发两次怎么办？→ 任务幂等 + 唯一执行 ID（8）。
- 任务还没跑完又到点了？→ 阻塞处理策略：串行/丢弃/覆盖（8）。
