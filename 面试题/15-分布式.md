# 15 · 分布式

> 覆盖：CAP/BASE、幂等设计、分布式事务（2PC/3PC/TCC/本地消息表/可靠消息/Saga）、分布式 ID、分布式锁对比、一致性哈希。是与 [07-SpringCloud](./07-SpringCloud.md)、[13-Zookeeper](./13-Zookeeper.md) 配套的理论核心。

---

## 一、理论基础

### 1. CAP 定理 🔥

分布式系统三选二：

- **C（一致性）**：所有节点同一时刻数据一致。
- **A（可用性）**：每个请求都能得到（非错误）响应。
- **P（分区容错性）**：网络分区时系统仍能运行。

由于网络分区 **P 在分布式中必然存在**，所以实际是在 **C 与 A 之间取舍**：

- **CP**：Zookeeper、etcd、HBase（一致性优先，分区时牺牲可用）。
- **AP**：Eureka、Nacos（默认）、Cassandra（可用性优先，最终一致）。

### 2. BASE 理论 🔥

对 CAP 中 AP 的工程化延伸，追求**最终一致**：

- **Basically Available（基本可用）**：允许损失部分可用性（降级、限流）。
- **Soft state（软状态）**：允许中间状态存在。
- **Eventually consistent（最终一致）**：经过一段时间后达到一致。

> 大多数互联网系统选择 BASE（最终一致）而非强一致，以换取可用性和性能。

## 二、幂等设计

### 3. 如何设计接口幂等 🔥

**幂等**：同一请求执行一次和多次效果相同。适用于支付、下单、MQ 重复消费等。

常见方案：

1. **唯一索引/主键**：重复插入直接失败（最简单可靠）。
2. **去重表 / Token 机制**：请求前申请 token，提交时校验并删除 token（一次性）。
3. **状态机**：限制状态流转（如订单只能从“待支付”→“已支付”）。
4. **乐观锁（version）**：`update ... where version=?`。
5. **分布式锁**：同一资源串行处理。
6. **Redis SETNX**：用唯一业务 ID 标记已处理。

> 注意区分：查询、删除天然幂等；新增、更新需要做幂等控制。

## 三、分布式事务

### 4. 为什么需要分布式事务 🔥

跨多个数据库/服务的操作要保证原子性（如下单：订单库 + 库存库 + 账户库）。单机 ACID 事务无能为力。

### 5. 2PC（两阶段提交）🔥

- 阶段一**准备**：协调者问参与者能否提交，参与者锁资源并预执行，回复 yes/no。
- 阶段二**提交/回滚**：全 yes 则提交，否则回滚。
- 缺点：**同步阻塞**（资源全程锁定）、**协调者单点**、阶段二消息丢失导致**不一致**。

### 6. 3PC ⭐

在 2PC 基础上加 **CanCommit** 阶段并引入**超时机制**（参与者超时默认提交），减少阻塞和单点问题，但仍不能完全解决数据一致性，且更复杂，实际很少用。

### 7. TCC（Try-Confirm-Cancel）🔥

业务层面的两阶段，需自己实现三个方法：

- **Try**：预留资源（如冻结库存）。
- **Confirm**：确认提交（真正扣减）。
- **Cancel**：取消（解冻）。

特点：**性能好、不锁数据库资源**，但**业务侵入强**，要处理空回滚、悬挂、幂等。适合金融等对一致性和性能都要求高的场景。

### 8. 本地消息表 / 可靠消息最终一致 🔥

**最常用的最终一致方案**：

1. 业务操作和“待发送消息”写入**同一个本地事务**（同库），保证原子。
2. 定时任务/异步线程扫描消息表，投递到 MQ。
3. 下游消费成功后回调标记消息完成；失败重试。
4. 配合消费端幂等。

RocketMQ 的**事务消息**是这套思路的内建实现（见 [11-MQ](./11-消息队列MQ.md)）。

### 9. Saga 模式 ⭐

把长事务拆成多个本地事务，每个都有对应的**补偿操作**；某步失败则**逆序执行已完成步骤的补偿**。适合流程长、参与者多的业务（如旅游订单：订机票→订酒店→租车）。无资源锁定，最终一致。

### 10. 分布式事务方案选型 🔥

| 方案 | 一致性 | 性能 | 侵入 | 场景 |
| --- | --- | --- | --- | --- |
| 2PC/XA | 强 | 低 | 低 | 强一致、低并发 |
| TCC | 强（最终） | 高 | 高 | 金融、核心交易 |
| 本地消息表/事务消息 | 最终 | 高 | 中 | 大多数互联网业务 |
| Saga | 最终 | 高 | 中 | 长流程业务 |
| Seata AT | 最终 | 中 | **低** | 通用、快速接入 |

> 工程原则：**能不用分布式事务就不用**（拆分边界设计好、用最终一致替代强一致）。

## 四、分布式 ID 与一致性哈希

### 11. 分布式全局唯一 ID 方案 🔥

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| UUID | 本地生成、无依赖 | 无序、占空间、不利索引 |
| 数据库自增/号段 | 简单、趋势递增 | 依赖 DB、扩展性一般 |
| **Snowflake 雪花** | 高性能、趋势递增、本地生成 | 依赖时钟，**时钟回拨**问题 |
| Redis incr | 高性能 | 依赖 Redis |
| 美团 Leaf / 百度 UidGenerator | 生产级、高可用 | 需部署 |

**Snowflake 64 位**：`1 符号位 + 41 时间戳 + 10 机器ID + 12 序列号`。时钟回拨可用等待、备用 ID、或 Leaf 的解决方案处理。

### 12. 一致性哈希 ⭐

把节点和数据都映射到一个 `0~2^32` 的**哈希环**上，数据顺时针找到第一个节点。

- 优点：节点增减时**只影响相邻一小段数据**，迁移量小（普通取模 hash 在扩容时几乎全部 key 要迁移）。
- **虚拟节点**：每个物理节点映射多个虚拟节点，解决数据倾斜、负载不均。
- 用途：分布式缓存（Redis 分片）、负载均衡、分库分表。

## 高频追问清单

- CAP 为什么只能三选二，实际怎么选？→ P 必选，C/A 取舍（第 1 题）。
- 分布式事务你们怎么做的？→ 优先最终一致（本地消息表/事务消息），核心用 TCC/Seata（第 8/10 题）。
- 接口幂等怎么实现？→ 唯一索引 + token + 状态机（第 3 题）。
- 一致性哈希解决了什么？→ 扩缩容时减少数据迁移（第 12 题）。
