# 29 · DDD 领域驱动设计

> 覆盖：DDD 是什么与价值、战略设计（限界上下文/通用语言）、战术设计（实体/值对象/聚合根/领域服务/仓储）、贫血 vs 充血模型、CQRS、事件驱动、与微服务的关系。架构师高频题。

---

## 1. DDD 是什么，解决什么问题 🔥

DDD（Domain-Driven Design，领域驱动设计）是一套**应对复杂业务**的软件设计方法论。核心思想：**以业务领域为中心建模**，让代码结构和业务概念对齐，而不是以技术/数据库表为中心。

解决的问题：

- 传统**贫血模型 + 大 Service** 随业务膨胀变成"面条代码"，业务逻辑散落各处。
- DDD 通过领域建模把业务知识沉淀到领域模型里，让复杂业务**可维护、可演进**。

> 注意：DDD 适合**复杂业务**，简单 CRUD 用 DDD 是过度设计。面试要点出这个"适用边界"。

---

## 2. 战略设计 🔥

宏观层面，划分业务边界：

- **领域 / 子域**：核心域（最有价值、要重点投入）、支撑域、通用域。
- **限界上下文（Bounded Context）**：一个领域模型的明确边界，同一个词在不同上下文含义不同（如"商品"在交易和库存上下文里不一样）。**限界上下文是微服务拆分的重要依据**。
- **通用语言（Ubiquitous Language）**：业务和技术团队用同一套术语，消除沟通歧义，代码命名也用这套语言。
- **上下文映射（Context Map）**：上下文之间的协作关系（防腐层 ACL、共享内核等）。

---

## 3. 战术设计 🔥（核心概念）

微观层面，领域模型的构建块：

| 概念 | 说明 |
| --- | --- |
| **实体（Entity）** | 有唯一标识、有生命周期（如订单、用户） |
| **值对象（Value Object）** | 无标识、不可变、用属性判等（如地址、金额） |
| **聚合（Aggregate）** | 一组相关实体/值对象的边界，作为一个整体被操作 |
| **聚合根（Aggregate Root）** | 聚合的入口和守门人，外部只能通过它访问聚合内部，保证**不变性/一致性** |
| **领域服务（Domain Service）** | 不属于单个实体的领域逻辑（如转账涉及两个账户） |
| **领域事件（Domain Event）** | 领域中发生的有意义的事（如"订单已支付"） |
| **仓储（Repository）** | 聚合的持久化抽象，屏蔽数据库细节 |
| **工厂（Factory）** | 创建复杂聚合 |

### 聚合设计原则 ⭐

- 聚合要**小**（一个聚合一个事务一致性边界）。
- 聚合之间通过**ID 引用**，不直接持有对象引用。
- 跨聚合的一致性用**领域事件 + 最终一致**，而非一个大事务。

## 4. 贫血模型 vs 充血模型 🔥

- **贫血模型（传统）**：领域对象只有 getter/setter（数据），业务逻辑全在 Service 层。简单直接，但业务复杂时 Service 臃肿、逻辑分散。
- **充血模型（DDD）**：业务逻辑**内聚到领域对象**自身（如 `order.pay()`、`account.withdraw()`），对象既有数据又有行为。更符合面向对象、业务表达力强。

```java
// 贫血：逻辑在 Service
orderService.pay(order); // order 只是数据

// 充血：逻辑在领域对象
order.pay(); // 校验状态、改状态、发领域事件 都在 Order 内部
```

## 5. DDD 分层架构 ⭐

```
用户接口层（Controller） → 应用层（Application Service，编排、事务、不含业务规则）
→ 领域层（Domain，核心业务模型与规则）→ 基础设施层（持久化、消息、外部接口）
```

- **应用层**很薄，只做编排（调领域对象、管事务），不写业务规则。
- **领域层**是核心，包含实体、聚合、领域服务、领域事件。
- 依赖倒置：领域层定义 Repository 接口，基础设施层实现（领域不依赖技术细节）。

---

## 6. CQRS（命令查询职责分离）⭐

把**写（Command）**和**读（Query）**分成不同模型：

- 写模型：领域模型，保证业务一致性。
- 读模型：为查询优化（可以是宽表/ES/缓存），不走领域模型。
- 适合**读写差异大、查询复杂**的场景。可配合**事件溯源（Event Sourcing）**：以事件序列作为唯一事实源，状态由事件重放得到。
- 代价：复杂度高、读写最终一致，按需使用。

---

## 7. DDD 与微服务的关系 🔥

- **限界上下文 ≈ 微服务边界**：DDD 帮你科学地拆分微服务（按业务能力而非技术分层），避免拆得过细或边界混乱。
- 微服务是 DDD 战略设计的一种落地形态，领域事件天然适合微服务间的异步解耦。
