# 32 · 操作系统

> 后端工程师的「共同地基」——很多组件高性能的原理都落到 OS 这一层。覆盖：进程/线程/协程、上下文切换、虚拟内存与 Page Cache、零拷贝、IO 模型与 epoll、CPU 缓存与内存屏障、用户态/内核态、死锁。读懂这篇，JVM/Netty/Kafka/Redis/Nginx 的「为什么快」才真正闭环。

## 一、进程、线程、协程

### 1. 进程 vs 线程 🔥

| | 进程 | 线程 |
| --- | --- | --- |
| 定义 | 资源分配的基本单位 | CPU 调度的基本单位 |
| 内存 | 独立地址空间 | 共享所属进程的地址空间 |
| 通信 | IPC（管道/消息队列/共享内存/Socket） | 共享内存，直接读写（需同步） |
| 开销 | 创建/切换开销大 | 小 |
| 隔离/稳定 | 强，一个崩不影响其它 | 弱，一个崩可能拖垮整个进程 |

### 2. 协程（Coroutine）⭐

- **用户态**的轻量级"线程"，由程序/运行时调度（不进内核），切换成本远低于线程。
- 一个线程可跑成千上万协程，遇到 IO 阻塞时**主动让出**，用同步写法实现异步性能。
- Java 的对应物是 **虚拟线程（JDK 21）**（见 [04-并发编程](./04-并发编程.md)）；Go 的 goroutine、Kotlin 协程同理。

### 3. 上下文切换（Context Switch）🔥

CPU 从一个执行流切到另一个时，要**保存当前寄存器/PC/栈等现场、加载新的现场**，这就是上下文切换，是**纯开销**。

- 触发：时间片用完、阻塞（IO/锁）、更高优先级抢占、系统调用。
- **线程切换**比进程切换便宜（不用换地址空间/TLB）；**协程切换**最便宜（用户态，不进内核）。
- 高并发下减少切换：用 IO 多路复用（少线程管多连接）、协程/虚拟线程、无锁化、合理线程池大小。

### 4. 用户态与内核态 🔥

- CPU 分**用户态（受限）** 和**内核态（特权，可操作硬件）**。
- 应用要做 IO、内存分配等需通过**系统调用（syscall）** 陷入内核态，完成后切回——这个切换有成本。
- **减少系统调用/态切换**是性能优化的常见方向（如批量 IO、用户态协议栈、零拷贝减少拷贝和切换）。

## 二、内存管理

### 5. 虚拟内存 🔥

每个进程拥有独立的**虚拟地址空间**，由 MMU 通过**页表**映射到物理内存。

- 好处：进程隔离（互不可见）、地址空间比物理内存大（配合换页）、便于共享（共享库/共享内存）。
- **缺页中断（Page Fault）**：访问的页不在物理内存时触发，OS 从磁盘换入（major fault）或分配新页（minor fault）。频繁缺页/换页（swap）会严重拖慢系统。

### 6. Page Cache（页缓存）🔥

OS 把磁盘文件数据缓存在内核内存中，读命中直接返回、写先写缓存再异步刷盘。

- **Kafka、ES、MySQL 都重度依赖 Page Cache** 提速：Kafka 顺序写 + Page Cache，几乎不用自己管缓存，重启后缓存还在内核（进程级缓存重启就没了）。
- 这也是为什么"给 Kafka/ES 机器留足空闲内存给 Page Cache"很重要。

### 7. 零拷贝（Zero-Copy）🔥🔥（高频）

传统「读文件→发网络」要 **4 次拷贝 + 4 次态切换**：磁盘→内核 Page Cache→用户缓冲区→内核 Socket 缓冲区→网卡。零拷贝去掉用户态的来回拷贝：

| 技术 | 原理 | 拷贝次数 |
| --- | --- | --- |
| `mmap + write` | 内存映射，用户态与内核共享 Page Cache | 减 1 次 |
| **`sendfile`** | 数据在内核内直接从文件传到 Socket，不经用户态 | 2~3 次 |
| `sendfile + DMA gather` | 连描述符都不拷贝，真正 0 次 CPU 拷贝 | CPU 0 次 |
| `splice` | 管道在内核内移动数据 | — |

- **Kafka** 用 `sendfile`（`FileChannel.transferTo`）把日志直接发给消费者。
- **Netty** 的 `FileRegion`、`CompositeByteBuf` 是零拷贝（见 [28-Netty](./28-Netty与网络编程.md)）。
- **RocketMQ** 用 `mmap`。

> 一句话：零拷贝 = **减少内核态↔用户态之间的数据拷贝和 CPU 参与**，是高吞吐中间件的标配。

## 三、IO 模型

### 8. 五种 IO 模型 🔥

以一次网络读为例（等数据就绪 + 拷贝数据两个阶段）：

1. **阻塞 IO（BIO）**：两个阶段都阻塞。
2. **非阻塞 IO**：轮询问"好了没"，没好就返回，忙等浪费 CPU。
3. **IO 多路复用**：一个线程用 select/poll/epoll 同时监听多个 fd，有就绪的才处理（**最常用**）。
4. **信号驱动 IO**：数据就绪发信号通知。
5. **异步 IO（AIO）**：两个阶段都由内核完成后回调通知，应用全程不阻塞。

> 前四种都是**同步 IO**（第二阶段拷贝数据时要应用参与/阻塞），只有 AIO 是真异步。

### 9. select / poll / epoll 的区别 🔥🔥

| | select | poll | **epoll** |
| --- | --- | --- | --- |
| fd 上限 | 1024（fd_set） | 无硬限制 | 无 |
| 数据结构 | 数组 | 链表 | 红黑树 + 就绪链表 |
| 效率 | 每次 O(n) 遍历全部 fd | 同样 O(n) | **O(1)**，只返回就绪 fd |
| fd 拷贝 | 每次调用全量拷贝到内核 | 同样 | `epoll_ctl` 注册一次，无需重复拷贝 |

- **epoll 高效的本质**：内核用回调把就绪 fd 放入就绪链表，`epoll_wait` 只返回就绪的，**避免遍历全部 fd**；fd 集合常驻内核无需反复拷贝。
- **Redis、Nginx、Netty(Linux)** 都基于 epoll。

### 10. 水平触发（LT）vs 边缘触发（ET）⭐

- **LT（默认）**：只要 fd 还有数据可读就**持续通知**，编程简单（没读完下次还会通知）。
- **ET**：状态变化时**只通知一次**，必须一次把数据读干净（循环读到 `EAGAIN`），效率高但易漏读。
- Nginx 用 ET，Netty/Redis 多用 LT。

## 四、CPU 与并发底层

### 11. CPU 缓存与缓存一致性（MESI）🔥

- CPU 有 L1/L2/L3 多级缓存，速度远快于内存；以**缓存行（Cache Line，通常 64 字节）** 为单位加载。
- 多核各有缓存，靠 **MESI 协议**（Modified/Exclusive/Shared/Invalid）保证缓存一致：一个核改了某缓存行，会让其它核的该行失效。
- **伪共享（False Sharing）**：不同线程改的变量落在同一缓存行，互相失效拖慢性能 → 缓存行填充 / `@Contended`（见 [04-并发编程](./04-并发编程.md)）。

### 12. 内存屏障与 happens-before ⭐

- CPU/编译器会**重排序**指令优化性能，多线程下导致可见性/有序性问题。
- **内存屏障（Memory Barrier）** 禁止特定重排、强制刷新缓存，是 `volatile`/锁底层的实现手段。
- Java 的 **JMM + happens-before** 就是对这套硬件语义的抽象（见 [04-并发编程](./04-并发编程.md) JMM）。

## 五、其它高频

### 13. 死锁的四个必要条件与处理 ⭐

互斥、请求并保持、不可剥夺、循环等待（与 Java 死锁一致，见 [04-并发编程](./04-并发编程.md)）。处理：预防（破坏条件）、避免（银行家算法）、检测 + 恢复。

### 14. 进程间通信（IPC）方式 ⭐

管道、命名管道、信号、消息队列、**共享内存（最快）**、信号量、Socket（可跨机）。

### 15. 磁盘 IO 与顺序写 ⭐

机械盘随机 IO 慢（寻道），**顺序写远快于随机写**。Kafka/数据库 WAL（redo log）都用**顺序追加写**换高吞吐（见 [09-MySQL](./09-MySQL.md) redo log、[33-Kafka](./33-Kafka.md)）。
