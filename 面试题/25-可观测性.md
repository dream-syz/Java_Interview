# 25 · 可观测性（监控 / 日志 / 链路追踪）

> 覆盖：可观测性三支柱（Metrics / Logging / Tracing）、Prometheus + Grafana、ELK 日志体系、链路追踪（SkyWalking/Zipkin）、告警、线上排障方法论。线上运维与排障是资深工程师的硬指标。

---

## 一、可观测性三支柱 🔥

| 支柱 | 回答的问题 | 典型工具 |
| --- | --- | --- |
| **Metrics（指标）** | 系统现在"健不健康"（QPS、RT、错误率、CPU、内存） | Prometheus + Grafana、Micrometer |
| **Logging（日志）** | "发生了什么"（详细事件、异常堆栈） | ELK / EFK、Loki |
| **Tracing（链路）** | "一个请求经过了哪些服务、慢在哪" | SkyWalking、Zipkin、Jaeger |

> 监控告诉你**有没有问题**，链路告诉你**问题在哪个环节**，日志告诉你**具体原因**。三者配合才能快速定位。

## 二、Metrics：Prometheus + Grafana 🔥

### 1. Prometheus 原理

- **拉模式（Pull）**：Prometheus 定时从各 target 的 `/metrics` 端点**抓取**指标（区别于 push）。
- **时序数据库（TSDB）**：按时间序列存储 `指标名{标签} 值 时间戳`。
- **PromQL**：强大的查询语言，做聚合、计算速率（`rate()`）等。
- **服务发现**：自动发现 K8s Pod / Consul 等 target。
- **Pushgateway**：短生命周期任务（批处理）主动推送的中转。

### 2. 四种指标类型

- **Counter（计数器）**：只增不减（请求总数）。
- **Gauge（仪表盘）**：可增可减（内存、在线人数）。
- **Histogram（直方图）**：分桶统计（RT 分布），可算分位数。
- **Summary（摘要）**：客户端算分位数。

### 3. Java 集成

Spring Boot Actuator + **Micrometer**（指标门面，类比 SLF4J）暴露 `/actuator/prometheus`，Prometheus 抓取，Grafana 展示大盘。

### 4. RED / USE 方法论

- **RED**（面向请求/服务）：**R**ate（请求率）、**E**rrors（错误率）、**D**uration（耗时）。
- **USE**（面向资源）：**U**tilization（使用率）、**S**aturation（饱和度）、**E**rrors。

## 三、Logging：ELK / EFK 日志体系 🔥

### 5. ELK 架构

```
应用日志 → Filebeat(采集) → Kafka(缓冲,可选) → Logstash(过滤/解析) → Elasticsearch(存储/检索) → Kibana(可视化)
```

- **Beats/Filebeat**：轻量采集器，部署在应用机器。
- **Kafka**：削峰缓冲，防日志洪峰压垮 ES（大流量必加）。
- **Logstash**：解析、格式化、富化（也可用更轻的 Ingest Pipeline）。
- **Elasticsearch**：存储 + 全文检索（见 [21-ES](./21-ElasticSearch.md)）。
- **Kibana**：查询、可视化、告警。
- **EFK**：用 Fluentd 替代 Logstash，K8s 环境常用。
- 轻量替代：**Grafana Loki**（只索引标签，成本低）。

### 6. 日志规范（实战）

- **统一格式**（JSON 结构化），含 `traceId`、`时间`、`级别`、`服务名`、`线程`。
- **traceId 贯穿全链路**（MDC + 拦截器），把分散日志串起来。
- 合理分级（ERROR 要告警，避免误用）、异步日志（Log4j2 AsyncLogger 性能高）、日志脱敏（手机号/身份证）。

## 四、Tracing：分布式链路追踪 🔥

### 7. 为什么需要

微服务一个请求跨多个服务，出问题难定位。链路追踪记录请求经过的每个节点和耗时，画出调用链。

### 8. 核心概念（OpenTracing/OpenTelemetry）

- **Trace**：一次完整请求的调用链（一个 traceId）。
- **Span**：链路中的一个工作单元（一次方法/RPC 调用），有 spanId、父 spanId。
- **traceId 传递**：通过 HTTP Header / RPC 上下文在服务间透传。
- **采样**：全量采集成本高，按比例采样。

### 9. 工具对比

| 工具 | 特点 |
| --- | --- |
| **SkyWalking** | 国产，**字节码增强无侵入**（agent），功能全（链路+指标+告警），主流 |
| Zipkin | 轻量、老牌、社区成熟 |
| Jaeger | CNCF，云原生 |
| **OpenTelemetry** | 标准规范（统一了 Trace/Metrics/Logs 采集），未来方向 |

SkyWalking 用 **Java Agent + 字节码增强**，无需改业务代码即可埋点（对比 Sleuth 需引依赖）。

## 五、线上排障方法论 🔥（资深必问）

### 10. 通用排障链路

1. **监控告警**发现异常（RT 飙升 / 错误率高 / CPU 满）。
2. **看大盘**（Grafana）定位异常服务和时间点。
3. **看链路**（SkyWalking）定位慢/错的具体环节。
4. **查日志**（Kibana，按 traceId）看具体异常堆栈。
5. **JVM 层**（Arthas / jstack / jmap）深入定位代码（见 [03-JVM](./03-JVM.md)）。

### 11. Arthas 实战命令 ⭐

```bash
dashboard          # 整体面板（线程/内存/GC）
thread -n 3        # 最忙的 3 个线程（排查 CPU 高）
thread -b          # 找死锁
trace Class method  # 方法内部各步骤耗时（排查慢方法）
watch Class method '{params,returnObj}'  # 观察入参/返回
jad Class          # 反编译看线上实际代码
ognl '@App@field'  # 运行时查看/修改值
profiler start/stop # 生成火焰图
```
